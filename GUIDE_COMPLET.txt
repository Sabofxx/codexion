=== QU'EST-CE QU'UN THREAD ? ===

    Imagine ton programme comme un restaurant. Normalement, t'as UN SEUL
    cuisinier (le thread principal) qui fait tout a la suite : couper les
    legumes, cuire la viande, dresser l'assiette. C'est SEQUENTIEL.

    Avec les threads, t'embauches 5 cuisiniers qui travaillent EN MEME TEMPS
    dans la meme cuisine. Chacun est un thread. Ils partagent le meme
    espace de travail (la memoire du programme), les memes couteaux et
    casseroles (les variables), mais ils avancent chacun a leur rythme.

    Un thread, c'est donc : un bout de code qui s'execute en parallele
    des autres, dans le meme programme, avec acces a la meme memoire.

    En C, on utilise la librairie pthread (POSIX threads) :
    - pthread_create() → cree un nouveau thread (embaucher un cuisinier)
    - pthread_join()   → attend qu'un thread finisse (attendre que le
                         cuisinier ait fini son service)

    Chaque thread execute une "routine" : une fonction qu'on lui donne
    au moment de la creation. Quand la fonction return, le thread meurt.


=== LE PROBLEME : LES RACE CONDITIONS ===

    Si deux cuisiniers essaient de couper sur LA MEME planche en meme
    temps, c'est le bordel. En programmation, c'est pareil.

    Exemple concret dans codexion :
    - Le coder (thread A) ecrit : compiles_done = compiles_done + 1
    - Le monitor (thread B) lit : if (compiles_done >= 3)

    Sans protection, voila ce qui peut arriver :
    1. Thread A lit compiles_done → voit 2
    2. Thread B lit compiles_done → voit 2 (AUSSI, avant que A ait fini)
    3. Thread A ecrit compiles_done = 3
    4. Thread B decide "2 < 3, pas fini" → FAUX ! C'est deja 3 !

    Ou pire :
    1. Thread A lit compiles_done → voit 2
    2. Thread A calcule 2 + 1 = 3
    3. Thread B lit compiles_done → voit 2 (pas encore ecrit !)
    4. Thread A ecrit 3
    5. Thread B ecrit 3 (il avait aussi calcule 2 + 1)
    → Resultat : 3 au lieu de 4. On a PERDU un compile.

    C'est ca une RACE CONDITION : le resultat depend de l'ordre d'execution
    des threads, qui est IMPREVISIBLE. Le programme marche "parfois" et
    plante "parfois". Le pire type de bug.


=== LA SOLUTION : LES MUTEX ===

    Un mutex (mutual exclusion = exclusion mutuelle), c'est un CADENAS.

    Imagine un WC avec un verrou. Une seule personne peut entrer a la fois.
    Les autres attendent devant la porte. Quand la personne sort et
    deverrouille, le suivant entre.

    En code :
        pthread_mutex_lock(&cadenas);     // j'entre et je verrouille
        // ... je fais mon truc tranquille, personne peut entrer ...
        compiles_done = compiles_done + 1;
        pthread_mutex_unlock(&cadenas);   // je sors et je deverrouille

    Les regles d'or :
    1. TOUJOURS lock avant de lire ou ecrire une variable partagee
    2. TOUJOURS unlock apres (sinon les autres attendent pour toujours)
    3. Garder la section "lockee" la plus COURTE possible (ne pas
       faire dormir le thread avec le cadenas pris)

    Si un thread appelle lock() et que le mutex est deja locke par un autre,
    le thread se MET EN PAUSE automatiquement. Il ne consomme pas de CPU,
    il dort. Quand le mutex est unlocke, il se reveille et prend le lock.


=== LES CONDITION VARIABLES (la sonnette) ===

    Probleme : un coder veut un dongle, mais il est pris. Il fait quoi ?

    Option 1 : il check en boucle "c'est libre ? non. c'est libre ? non."
    → C'est du BUSY WAITING. Ca bouffe du CPU pour rien. C'est nul.

    Option 2 : il se couche et dit "reveillez-moi quand ca change".
    → C'est une CONDITION VARIABLE. C'est ce qu'on utilise.

    Une condition variable, c'est une SONNETTE reliee a un cadenas (mutex).
    - pthread_cond_wait(sonnette, cadenas)   → je m'endors
      En UNE seule operation atomique, ca fait :
      1. Libere le cadenas (pour que d'autres puissent changer l'etat)
      2. S'endort
      3. Quand quelqu'un sonne → se reveille et reprend le cadenas
    - pthread_cond_broadcast(sonnette) → sonne pour reveiller TOUT le monde
    - pthread_cond_signal(sonnette)    → reveille UN SEUL dormeur (aleatoire)

    ATTENTION : quand tu te reveilles, ca ne veut PAS dire que la condition
    est vraie ! Quelqu'un d'autre a pu passer avant toi. C'est pour ca
    qu'on utilise TOUJOURS une boucle while, jamais un if :

        while (!condition_remplie)      // ← WHILE, pas IF !
            pthread_cond_wait(&cond, &mutex);

    Ca s'appelle un "spurious wakeup check" : a chaque reveil, on re-verifie.

    pthread_cond_timedwait c'est pareil mais avec un timeout : si personne
    sonne avant X millisecondes, le thread se reveille quand meme. C'est un
    filet de securite pour ne pas rester coince.


=== LE DEADLOCK (l'ennemi mortel) ===

    Un deadlock, c'est quand tout le monde attend tout le monde et PLUS
    PERSONNE ne peut avancer. Le programme est gele pour toujours.

    Exemple classique (le "probleme des philosophes") :
    - 5 personnes a une table ronde
    - 5 fourchettes entre elles
    - Pour manger, il faut 2 fourchettes (gauche et droite)
    - Tout le monde prend sa fourchette gauche en meme temps
    - Tout le monde attend la fourchette droite
    - Personne ne mange. Personne ne lache. DEADLOCK.

    Codexion, c'est exactement ce probleme mais avec des coders et des
    dongles (cles USB) au lieu de philosophes et de fourchettes.

    Il y a 4 conditions pour un deadlock (les conditions de Coffman) :
    1. Exclusion mutuelle : une ressource ne peut etre utilisee que par un
    2. Hold and wait : on tient une ressource en attendant une autre
    3. No preemption : on ne peut pas forcer quelqu'un a lacher
    4. Circular wait : A attend B, B attend C, C attend A → cycle

    Pour eviter le deadlock, il SUFFIT de casser UNE de ces 4 conditions.
    Dans codexion, on casse le CIRCULAR WAIT. Comment ? En imposant un
    ORDRE : chaque coder prend toujours le dongle de plus petit ID en premier.
    (On expliquera ca en detail dans dongle.c)


=== RESUME DES ANALOGIES POUR CODEXION ===

    Thread  = un employe qui travaille en parallele des autres
    Mutex   = un cadenas / verrou de porte (1 seul entre a la fois)
    Cond    = une sonnette (reveille ceux qui dorment quand ca change)
    Dongle  = une cle USB partagee (comme une fourchette chez les philosophes)
    Coder   = un philosophe / employe qui a besoin de 2 dongles
    Monitor = un surveillant qui check si quelqu'un est mort
    Burnout = la mort d'un philosophe (il n'a pas mange/compile a temps)
    Deadlock= tout le monde bloque, plus personne ne bouge

─── LIGNES 13-14 : Include guard ───

    #ifndef CODEXION_H
    # define CODEXION_H

    C'est comme dire : "Est-ce que j'ai deja lu cette page ?"
    - La premiere fois, CODEXION_H n'existe pas → on definit CODEXION_H
      et on lit tout le fichier
    - Si un autre .c inclut codexion.h une 2eme fois → CODEXION_H existe
      deja → on saute tout le contenu → pas de doublon

    Sans ca, le compilateur verrait les memes structs deux fois et dirait
    "erreur : type deja defini". C'est obligatoire dans tout header.

─── LIGNES 23-27 : Enum t_sched ───

    typedef enum e_sched
    {
        SCH_FIFO,     → vaut automatiquement 0
        SCH_EDF       → vaut automatiquement 1
    }   t_sched;

    Un "enum" (enumeration) c'est donner des NOMS a des nombres.
    Au lieu d'ecrire "if (scheduler == 0)" (c'est quoi 0 ??),
    on ecrit "if (scheduler == SCH_FIFO)" → on comprend tout de suite.

    FIFO = First In, First Out = "premier arrive, premier servi".
    Imagine une file d'attente a la boulangerie : celui qui arrive en premier
    est servi en premier.

    EDF = Earliest Deadline First = "le plus urgent passe en premier".
    Imagine les urgences a l'hopital : on traite d'abord celui qui va mourir
    le plus tot, meme s'il est arrive en dernier.

    Note technique : on a du renommer de "SCHED_FIFO" a "SCH_FIFO" parce
    que macOS definit deja SCHED_FIFO dans ses headers systeme. Si on garde
    le meme nom → conflit → erreur de compilation.

─── LIGNES 29-33 : Struct t_request ───

    typedef struct s_request
    {
        int     coder_id;   → quel coder a fait cette demande (1..N)
        long    key;        → cle de priorite (plus petit = plus urgent)
    }   t_request;

    Imagine que tu prends un ticket a la boulangerie. Sur le ticket il y a :
    - Ton nom (coder_id)
    - Un numero (key)

    La key depend du mode de scheduling :
    - FIFO : key = l'heure exacte a laquelle tu as pris le ticket
      → le ticket avec l'heure la plus ancienne est servi en premier
    - EDF  : key = l'heure a laquelle tu vas "mourir" (burnout)
      → le ticket avec la deadline la plus proche est servi en premier

    Dans LES DEUX cas : plus petite key = plus haute priorite. C'est
    la beaute du systeme : on change juste comment on calcule la key,
    et tout le reste du code fonctionne pareil.

─── LIGNES 35-41 : Struct t_pq (Priority Queue) ───

    typedef struct s_pq
    {
        t_request   *data;   → tableau dynamique de tickets
        int         size;    → combien de tickets dans la file actuellement
        int         cap;     → combien de tickets la file peut contenir max
        int         (*cmp)(t_request, t_request);  → comment comparer 2 tickets
    }   t_pq;

    C'est notre FILE DE PRIORITE, implementee comme un MIN-HEAP
    (on explique en detail dans priority_queue.c).

    En gros : c'est une boite ou tu mets des tickets, et quand tu demandes
    "c'est qui le prochain ?", elle te donne TOUJOURS celui avec la plus
    petite key. Pas besoin de chercher, c'est automatique.

    Le champ (*cmp) est un POINTEUR DE FONCTION. C'est bizarre au debut :
    c'est une variable qui contient... une fonction. Comme si tu stockais
    une recette dans un tiroir. Quand la pq a besoin de comparer deux
    tickets, elle appelle cette fonction. Ca permet a la pq de fonctionner
    avec n'importe quel type de comparaison sans la modifier.

    Pourquoi un heap et pas juste une liste triee ?
    → Ajouter un ticket : O(log N) avec un heap, O(N) avec une liste
    → Retirer le plus prioritaire : O(log N) dans les deux cas
    → Le heap est plus rapide. Et le sujet l'impose.

─── LIGNE 43 : Forward declaration ───

    typedef struct s_sim    t_sim;

    Petit probleme technique : t_dongle contient un pointeur vers t_sim,
    et t_sim contient un pointeur vers t_dongle. C'est circulaire :
    chacun a besoin que l'autre soit defini en premier.

    La solution : on dit au compilateur "t_sim existe, je te donne les
    details plus tard". C'est ca une forward declaration. Ca suffit pour
    declarer des POINTEURS vers t_sim (parce qu'un pointeur a toujours
    la meme taille, peu importe ce vers quoi il pointe).

─── LIGNES 45-55 : Struct t_dongle ───

    typedef struct s_dongle
    {
        int             id;           → numero du dongle (0, 1, 2...)
        int             in_use;       → 1 = quelqu'un le tient, 0 = libre
        int             inited;       → 1 = tout a ete initialise proprement
        long            released_at;  → quand (en ms) le dongle a ete repose
        long            cooldown;     → combien de ms attendre avant de le reprendre
        pthread_mutex_t lock;         → le cadenas qui protege cette struct
        pthread_cond_t  cond;         → la sonnette pour reveiller les dormeurs
        t_pq            waiters;      → la file d'attente des coders
    }   t_dongle;

    Un dongle, c'est comme une fourchette dans le probleme des philosophes.
    C'est une RESSOURCE PARTAGEE : plusieurs coders en ont besoin, mais UN
    SEUL peut l'utiliser a la fois.

    Detaillons chaque champ :

    "lock" (mutex) : c'est le cadenas qui protege TOUT dans cette struct.
    Avant de lire ou modifier quoi que ce soit (in_use, released_at,
    waiters...), un thread DOIT d'abord prendre ce lock. Sinon → race
    condition → bug.

    "cond" (condition variable) : c'est la sonnette reliee au cadenas.
    Quand un coder repose le dongle, il fait "broadcast" sur cond = il
    sonne. Tous les coders endormis (qui attendaient ce dongle) se
    reveillent et verifient si c'est leur tour.

    "released_at" + "cooldown" : quand un coder repose le dongle, on note
    l'heure. Le dongle ne peut etre repris que apres cooldown ms.
    Exemple : released_at = 5000, cooldown = 50 → reprenable a 5050.
    C'est comme un temps de refroidissement dans un jeu video.

    "waiters" (priority queue) : la file d'attente. Quand un coder veut
    le dongle mais ne peut pas l'avoir, il met son ticket ici. Quand le
    dongle se libere, c'est le coder avec le meilleur ticket qui passe.

    "inited" : un flag technique pour le nettoyage. Quand on free la
    memoire a la fin, on ne detruit que ce qui a vraiment ete initialise.
    Si l'init a plante a mi-chemin, on ne destroy pas ce qui n'existe pas.

─── LIGNES 57-66 : Struct t_coder ───

    typedef struct s_coder
    {
        int             id;             → numero affiche (1, 2, 3...)
        int             inited;         → 1 = mutex initialisee
        pthread_t       thread;         → identifiant du thread
        int             compiles_done;  → combien de compiles termines
        long            last_compile;   → quand (en ms) il a compile pour la derniere fois
        pthread_mutex_t mtx;            → cadenas pour proteger ses donnees
        t_sim           *sim;           → pointeur vers la simulation
    }   t_coder;

    Un coder = un thread = un employe qui tourne en boucle :
    prendre 2 dongles → compiler → poser les dongles → debugger →
    refactorer → recommencer.

    "thread" : c'est le numero d'identite du thread cree par pthread_create.
    On en a besoin pour pthread_join (attendre qu'il finisse).

    "compiles_done" : le compteur. Quand il atteint compiles_req (le nombre
    demande dans les arguments), le coder a fini sa mission.

    "last_compile" : l'heure du dernier repas (comme pour les philosophes).
    Si maintenant - last_compile >= t_burnout → le coder est mort (burnout).
    C'est le monitor qui surveille cette valeur en permanence.

    "mtx" : le cadenas du coder. Protege compiles_done et last_compile.
    POURQUOI ? Parce que ces deux champs sont lus par le monitor (un
    autre thread) et ecrits par le coder (son propre thread). Si deux
    threads lisent/ecrivent en meme temps sans cadenas → les valeurs
    peuvent etre corrompues → faux burnout ou faux "tout le monde a fini".

    "sim" : un pointeur vers la structure mere. Via c->sim, le coder
    accede a tout : les dongles, la config, le flag stop, etc.

    "id" est 1-indexe pour l'affichage ("1 is compiling", pas "0 is
    compiling"). En interne, le coder d'id 1 est stocke dans sim->coders[0].

─── LIGNES 68-83 : Struct s_sim (la structure mere) ───

    struct s_sim
    {
        int             n_coders;       → nombre de coders (et de dongles)
        long            t_burnout;      → temps en ms avant burnout
        long            t_compile;      → duree d'un compile en ms
        long            t_debug;        → duree d'un debug en ms
        long            t_refactor;     → duree d'un refactor en ms
        int             compiles_req;   → combien de compiles chaque coder doit faire
        long            cooldown;       → temps entre deux utilisations du meme dongle
        t_sched         sched;          → FIFO ou EDF ?
        long            start_ms;       → timestamp de reference (le "zero")
        int             stop;           → 1 = c'est fini, tout le monde arrete
        pthread_mutex_t log_mtx;        → cadenas pour les logs et stop
        pthread_t       monitor;        → le thread surveillant
        t_dongle        *dongles;       → tableau de N dongles
        t_coder         *coders;        → tableau de N coders
    };

    C'est le "cerveau" du programme. Tout est la-dedans.
    Chaque coder a un pointeur vers sim (c->sim), donc il peut acceder
    a tout : les dongles, les timings, le flag stop...

    "stop" : le signal d'arret global. Quand stop passe a 1, TOUS les
    threads doivent s'arreter le plus vite possible. C'est comme une alarme
    incendie : quand elle sonne, tout le monde sort.

    "log_mtx" : LE mutex le plus important. Il protege DEUX choses :
    1. Le flag "stop" → pour que deux threads ne le modifient pas en
       meme temps
    2. L'acces a printf → pour que les messages ne se melangent pas

    Imagine 5 coders qui crient en meme temps. Sans cadenas, ca donne :
    "1 is com2 is depiling3 is..." → illisible. Avec le cadenas, chacun
    attend son tour pour parler.

    "start_ms" : l'heure zero. Tous les timestamps affiches sont
    relatifs a cette valeur : temps_affiche = now_ms() - start_ms.
    Ca donne "0 ms", "42 ms", etc. au lieu de "1719847561234 ms".

    "monitor" : le thread surveillant qui tourne en boucle pour verifier
    si quelqu'un a fait un burnout ou si tout le monde a fini.

─── now_ms() ───  ligne 15

    long    now_ms(void)

    Retourne l'heure actuelle en MILLISECONDES.

    gettimeofday(&tv, NULL) → remplit une struct avec :
    - tv.tv_sec  = secondes depuis le 1er janvier 1970
    - tv.tv_usec = microsecondes restantes (0 a 999999)

    Calcul : tv.tv_sec * 1000 + tv.tv_usec / 1000
    - secondes × 1000 → convertit en millisecondes
    - microsecondes ÷ 1000 → convertit en millisecondes
    - on additionne

    Pourquoi en millisecondes ? Parce que tous les timings du sujet sont
    en ms (t_burnout, t_compile, etc.). C'est notre unite de base.
    On utilise cette fonction PARTOUT pour mesurer le temps.

─── precise_sleep() ───  ligne 24

    void    precise_sleep(long ms, t_sim *sim)

    Fait dormir le thread pendant "ms" millisecondes, mais en restant
    REACTIF au flag stop.

    Pourquoi pas juste usleep(ms * 1000) ?
    Parce que si un coder dort pour 200ms (compile time) et que quelqu'un
    fait un burnout a 50ms, il faudrait attendre 150ms de trop avant que
    le coder s'arrete. C'est inacceptable : le sujet veut de la precision.

    Algorithme :
    target = now_ms() + ms   → l'heure a laquelle on veut se reveiller
    while (now_ms() < target)
        if (sim_stopped(sim)) return     → si c'est fini, on sort TOUT DE SUITE
        usleep(200)                      → dort 0.2ms puis re-check

    C'est un COMPROMIS :
    - usleep(200) = 0.2ms de micro-siestes → on check stop tres souvent
    - Precision : on sort au plus tard 0.2ms apres le flag stop
    - CPU : on dort quand meme (usleep libere le CPU), pas du busy waiting

    C'est comme mettre un reveil toutes les 12 secondes quand tu fais une
    sieste de 20 minutes. Si l'alarme incendie sonne, tu reagis en moins
    de 12 secondes au lieu de dormir 20 minutes completes.

─── log_state() ───  ligne 35

    void    log_state(t_coder *c, const char *state)

    Affiche un message du type "42 5 is compiling\n".
    42 = millisecondes depuis le debut, 5 = id du coder.

    PROTECTION DU LOG :
    1. pthread_mutex_lock(&sim->log_mtx)   → prend le cadenas
    2. if (!sim->stop)                      → verifie que c'est pas fini
    3. printf(...)                           → affiche le message
    4. pthread_mutex_unlock(&sim->log_mtx) → lache le cadenas

    Pourquoi le check "if (!stop)" ?
    → Quand la simulation s'arrete (burnout ou all_done), on ne veut PLUS
    de messages apres le message d'arret. Sans ce check, un coder pourrait
    afficher "is debugging" APRES "burned out", ce qui est incoherent.

    Pourquoi le mutex ?
    → printf n'est PAS thread-safe. Si deux threads appellent printf en
    meme temps, les caracteres peuvent se melanger. Le mutex garantit
    qu'un seul thread parle a la fois.

    Le timestamp : now_ms() - sim->start_ms → temps relatif en ms.
    Comme ca l'affichage commence a 0, pas a 1719847561234.

─── sim_stopped() ───  ligne 48

    int     sim_stopped(t_sim *sim)

    Verifie si la simulation est arretee. Thread-safe.

    1. Lock log_mtx
    2. Lit sim->stop (0 ou 1)
    3. Unlock
    4. Retourne la valeur lue

    Pourquoi cette fonction ? Pourquoi pas juste lire sim->stop directement ?
    → Parce que sim->stop est modifie par le monitor et lu par les coders.
    Sans le mutex, on risque de lire une valeur "a moitie ecrite".
    OK en pratique pour un int, mais en theorie → undefined behavior.
    Et surtout : le mutex garantit la VISIBILITE. Sans lui, le compilateur
    ou le CPU pourrait cacher une ancienne valeur de stop dans un registre,
    et le thread ne verrait jamais que stop est passe a 1.

    On appelle cette fonction des DIZAINES de fois par seconde (dans
    precise_sleep, dans acquire_dongle...) pour reagir vite au stop.

                   |  t_burnout

─── parse_one() ───  ligne 15

    static long parse_one(const char *s)

    Convertit UNE string en nombre. "200" → 200.

    "static" signifie que cette fonction n'est visible QUE dans ce fichier.
    C'est comme une fonction "privee".

    Parcourt chaque caractere :
    1. Si un caractere n'est PAS un chiffre (pas entre '0' et '9') → -1
    2. Sinon : result = result * 10 + (c - '0')
       - c - '0' : astuce classique en C. Les caracteres '0' a '9' sont
         des nombres consecutifs dans la table ASCII. '5' - '0' = 5.
       - result * 10 : decale les chiffres deja lus vers la gauche.
         Exemple : "42" → result = 0 → 0*10 + 4 = 4 → 4*10 + 2 = 42

    Retourne -1 si :
    - Un caractere n'est pas un chiffre
    - Le resultat est 0 ou negatif (valeurs invalides pour nos parametres)

─── parse_numbers() ───  ligne 30

    static int  parse_numbers(int ac, char **av, t_sim *sim)

    Parse les 7 valeurs numeriques (args 1 a 7).

    Utilise parse_one() pour chacun, verifie qu'aucun n'est -1.
    Remplit : n_coders, t_burnout, t_compile, t_debug, t_refactor,
    compiles_req, cooldown.

    Note : n_coders est un int (nombre de personnes), les temps sont des
    long (millisecondes). On caste n_coders en int apres le parse.

─── parse_sched() ───  ligne 49

    static int  parse_sched(const char *s, t_sim *sim)

    Parse le 8eme argument : "fifo" ou "edf".

    strcmp(s, "fifo") == 0 → c'est FIFO
    strcmp(s, "edf") == 0  → c'est EDF
    sinon → erreur (return 0)

    strcmp retourne 0 quand les deux strings sont identiques. Oui,
    c'est bizarre. 0 = "zero difference" = identiques.

    1. Verifie le nombre d'arguments (8 ou 9 si compiles_req optionnel)
    2. parse_numbers → les 7 nombres
    3. parse_sched → le scheduler

    Retourne 1 si OK, 0 si erreur.

Ce fichier prepare TOUTES les ressources avant de lancer les threads.

─── init_one_dongle() ───  ligne 15

    static int  init_one_dongle(t_dongle *d, int id, t_sim *sim)

    Initialise UN dongle (struct + mutex + cond + pq).

    1. memset(d, 0, sizeof(*d))
       → Met TOUT a zero. C'est propre : in_use=0, released_at=0, etc.
       memset c'est comme passer un coup d'eponge : on repart de zero.

    2. d->id = id, d->cooldown = sim->cooldown
       → Assigne le numero et le temps de cooldown

    3. pthread_mutex_init(&d->lock, NULL)
       → Cree le cadenas du dongle. NULL = options par defaut.
       Si ca echoue → return 0 (erreur systeme, tres rare)

    4. pthread_cond_init(&d->cond, NULL)
       → Cree la sonnette du dongle.
       Si ca echoue → destroy le mutex deja cree (cleanup) → return 0

    5. pq_init(&d->waiters, sim->n_coders, cmp_request)
       → Cree la file d'attente. Capacite = n_coders (au pire, tout
       le monde attend ce dongle). cmp_request = la fonction de
       comparaison qu'on veut utiliser.
       Si ca echoue → destroy la cond + le mutex → return 0

    6. d->inited = 1
       → "Ce dongle est pret." Le flag sert au cleanup : si l'init a
       plante au dongle 3, on ne detruira que les dongles 0, 1, 2.

    C'est un pattern classique : init en couches, et si une couche echoue,
    on defait les couches precedentes dans l'ordre inverse. Comme empiler
    des cartes : si la 3eme tombe, tu enleves la 3eme, la 2eme, la 1ere.

    static int  init_one_coder(t_coder *c, int idx, t_sim *sim)

    int     init_sim(t_sim *sim)

#   FICHIER 5 : srcs/destroy_sim.c  (le nettoyage)                            #
#                                                                              #
################################################################################

On free TOUT ce qui a ete alloue. Mutex, cond vars, priority queues,
tableaux... C'est l'inverse exact de init_sim.

─── destroy_one_dongle() ───  ligne 15

    static void destroy_one_dongle(t_dongle *d)

    Detruit un dongle SI il a ete initialise (d->inited == 1).

    Ordre de destruction = INVERSE de l'ordre de creation :
    1. pq_destroy → free le tableau de la file d'attente
    2. pthread_cond_destroy → detruit la sonnette
    3. pthread_mutex_destroy → detruit le cadenas

    Pourquoi l'ordre inverse ? Parce qu'on demonte de l'exterieur vers
    l'interieur. C'est comme defaire une poupee russe : on enleve la
    couche exterieure avant de toucher l'interieure.

─── destroy_sim() ───  ligne 27

    void    destroy_sim(t_sim *sim)

    1. Si dongles existe → destroy chaque dongle → free le tableau
    2. Si coders existe → destroy chaque mutex coder → free le tableau
    3. pthread_mutex_destroy(&sim->log_mtx) → le cadenas des logs

    Le pattern "if (ptr) { ... free(ptr) }" est obligatoire : si le
    malloc a echoue, ptr est NULL. Appeler free(NULL) est OK en C,
    mais acceder a ptr[i] quand ptr est NULL → crash.

─── set_start_time() ───  ligne 15

    static void set_start_time(t_sim *sim)

    Enregistre le moment exact du debut de la simulation.

    1. sim->start_ms = now_ms() → l'heure zero
    2. Pour chaque coder : last_compile = start_ms
       → Chaque coder "a compile" au temps zero. Ca demarre leur
       compteur de burnout. Sans ca, le monitor verrait
       "last_compile = 0, now = 1719847561234 → burnout instantane !"

    C'est tres important : on initialise last_compile JUSTE AVANT de
    lancer les threads. Si on le faisait trop tot, le temps entre l'init
    et le lancement des threads compterait comme du "temps sans compiler",
    et les coders pourraient mourir avant d'avoir eu une chance.

─── launch_threads() ───  ligne 24

    static int  launch_threads(t_sim *sim)

    Cree tous les threads.

    1. Pour chaque coder : pthread_create → lance un thread executant
       coder_routine, avec un pointeur vers le coder comme argument.
       Si echec → set stop = 1 (pour arreter ceux deja lances), return 0.

    2. pthread_create pour le monitor → lance monitor_routine.
       Si echec → stop = 1, return 0.

    pthread_create prend 4 arguments :
    - &c->thread → ou stocker l'identifiant du thread cree
    - NULL → options par defaut
    - coder_routine → la fonction que le thread va executer
    - c → l'argument passe a la fonction (void* = pointeur generique)

    A partir de cet instant, N+1 threads tournent en PARALLELE.
    Le main thread n'a plus rien a faire et attend.

─── join_threads() ───  ligne 43

    static void join_threads(t_sim *sim)

    Attend que TOUS les threads finissent.

    pthread_join(thread, NULL) → bloque jusqu'a ce que le thread termine.
    C'est comme un parent qui attend ses enfants a la sortie de l'ecole.
    Il ne part pas tant que tout le monde n'est pas sorti.

    On join les coders, puis le monitor. L'ordre n'a pas d'importance
    technique, mais c'est propre : on attend tout.

─── main() ───  ligne 55

    int     main(int ac, char **av)

    Le flow complet du programme en 5 lignes :

    1. parse_args → lire et valider les arguments
       Echec → "Error\n" + return 1
    2. init_sim → preparer toutes les ressources
       Echec → destroy + return 1
    3. set_start_time → noter l'heure zero
    4. launch_threads → lancer tout le monde
       Echec → join + destroy + return 1
    5. join_threads → attendre la fin
    6. destroy_sim → nettoyer
    7. return 0 → sortie propre

    Chaque etape verifie les erreurs et nettoie si necessaire. C'est un
    pattern "init or die" : si ca foire, on defait tout et on sort.

─── update_last_compile() ───  ligne 15

    static void update_last_compile(t_coder *c)

    Met a jour l'horodatage du dernier compile.

    1. Lock c->mtx
    2. c->last_compile = now_ms()
    3. Unlock

    Pourquoi sous mutex ? Le monitor lit last_compile pour checker les
    burnouts. Si le coder ecrit ET le monitor lit en meme temps → race
    condition → valeur corrompue → faux burnout.

    On appelle cette fonction JUSTE AVANT de compiler. C'est comme
    remettre le chronometre du burnout a zero. "Je suis vivant !"

─── get_last_compile() ───  ligne 24

    long    get_last_compile(t_coder *c)

    Lit last_compile de maniere thread-safe (sous mutex).
    Utilisee par le monitor ET par make_request (pour calculer la deadline EDF).

─── get_compiles_done() ───  ligne 33

    int     get_compiles_done(t_coder *c)

    Lit compiles_done de maniere thread-safe (sous mutex).
    Utilisee par le monitor pour verifier si tout le monde a fini.

─── inc_compiles_done() ───  ligne 42

    static void inc_compiles_done(t_coder *c)

    Incremente compiles_done de maniere thread-safe.

─── coder_routine() ───  ligne 50  ★ LE COEUR DU CODER ★

    void    *coder_routine(void *arg)

    C'est la fonction que chaque thread coder execute. Elle recoit un
    void* (pointeur generique) qu'on caste en t_coder*.

    La boucle infinie :

    while (!sim_stopped(sim))               ← tant que c'est pas fini
    {
        // ETAPE 1 : Prendre 2 dongles
        if (take_dongles(c))                ← prend les dongles (peut bloquer)
            break;                          ← si echec (sim arretee), on sort

        // ETAPE 2 : Compiler
        update_last_compile(c);             ← remet le chrono burnout a zero
        log_state(c, "is compiling");       ← affiche "X is compiling"
        precise_sleep(sim->t_compile, sim); ← dort pendant t_compile ms
        release_dongles(c);                 ← repose les 2 dongles
        inc_compiles_done(c);               ← +1 au compteur

        // ETAPE 3 : Debugger (ne necessite pas de dongle)
        if (sim_stopped(sim)) break;
        log_state(c, "is debugging");
        precise_sleep(sim->t_debug, sim);

        // ETAPE 4 : Refactorer (ne necessite pas de dongle)
        if (sim_stopped(sim)) break;
        log_state(c, "is refactoring");
        precise_sleep(sim->t_refactor, sim);
    }
    return (NULL);                          ← le thread meurt

    POINTS IMPORTANTS :

    1. On check sim_stopped AVANT chaque etape. Si la simu s'arrete pendant
       un sleep, on sort des que le sleep finit (grace a precise_sleep).

    2. Les dongles sont pris SEULEMENT pour compiler. Debug et refactor
       se font sans dongle → les dongles sont libres pour les autres.

    3. Le burnout est reset AVANT le compile, pas apres. Parce que le
       moment critique c'est L'ATTENTE des dongles. Si un coder attend
       500ms pour un dongle et que t_burnout = 400ms, il meurt pendant
       l'attente. Mettre update_last_compile apres le compile ne change
       rien pour ce cas → le burnout est pendant l'attente.

    4. Quand take_dongles retourne 1 (echec), c'est que sim_stopped est
       devenu vrai. On break sans essayer de release (on n'a rien pris).

─── get_dongle_ids() ───  ligne 15

    static void get_dongle_ids(t_coder *c, int *first, int *second)

    Calcule QUELS dongles le coder doit prendre, et dans QUEL ORDRE.
    C'est ici que se trouve la SOLUTION ANTI-DEADLOCK.

    Imagine la table ronde. Les coders sont assis en cercle, et entre
    chaque paire de coders il y a un dongle. Comme les philosophes avec
    leurs fourchettes.

    Le coder d'id=i (id commence a 1) a :
    - dongle gauche = i - 1  (index 0-base)
    - dongle droit  = i % N  (index 0-base, N = nombre total)

    Exemple concret avec 5 coders et 5 dongles :

        Dongle 0        Dongle 1        Dongle 2        Dongle 3        Dongle 4
           |                |                |                |               |
        Coder 1          Coder 2          Coder 3          Coder 4         Coder 5
       (left=0,right=1) (left=1,right=2) (left=2,right=3) (left=3,right=4) (left=4,right=0)

    Pour les coders 1 a 4 : left < right → first = left, second = right
    Coder 1 : first=0, second=1  ← prend le plus petit d'abord ✓
    Coder 2 : first=1, second=2  ✓
    Coder 3 : first=2, second=3  ✓
    Coder 4 : first=3, second=4  ✓

    Pour le coder 5 : left=4, right=0 → left > right → INVERSION !
    Coder 5 : first=0, second=4  ← prend 0 AVANT 4 !

    C'est cette inversion qui casse le CIRCULAR WAIT :

    SANS l'inversion (tout le monde prend sa gauche d'abord) :
    → Coder 1 prend dongle 0, attend dongle 1
    → Coder 2 prend dongle 1, attend dongle 2
    → Coder 3 prend dongle 2, attend dongle 3
    → Coder 4 prend dongle 3, attend dongle 4
    → Coder 5 prend dongle 4, attend dongle 0 ← BOUCLE ! Dongle 0 est
      tenu par Coder 1 qui attend Coder 2 qui attend... → DEADLOCK !

    AVEC l'inversion (le coder 5 prend dongle 0 d'abord) :
    → Coder 5 essaie de prendre dongle 0, MAIS Coder 1 le veut aussi
    → L'un des deux attend → l'autre avance → pas de cycle → PAS DE DEADLOCK

    Regle generale : prendre toujours le dongle de plus petit ID en premier.
    Ca cree un ORDRE TOTAL et rend les cycles impossibles.

    Cas special N=1 : left = 0, right = 0 → first = second = 0.
    Le coder a besoin du MEME dongle deux fois → impossible de compiler.

─── wait_for_stop() ───  ligne 34

    static int  wait_for_stop(t_coder *c)

    Gere le cas special N=1. Le coder ne pourra JAMAIS compiler
    (il faut 2 dongles, il n'y en a qu'un). Il attend simplement que
    le monitor detecte le burnout et arrete la simulation. Retourne 1.

    C'est un cas degenere : la simulation demarre, le coder attend,
    le timer de burnout s'ecoule, le monitor dit "burned out", fin.

─── take_dongles() ───  ligne 41

    int     take_dongles(t_coder *c)

    Prend les 2 dongles du coder. Retourne 0 si OK, 1 si echec.

    Flow etape par etape :

    1. get_dongle_ids(c, &first, &second)
       → Calcule quel dongle prendre en premier et en deuxieme

    2. Si first == second → N=1, on appelle wait_for_stop

    3. acquire_dongle(dongle[first], c)
       → Essaie de prendre le PREMIER dongle (peut bloquer longtemps)
       → Si echec (simulation arretee pendant l'attente) → return 1
       → Si OK → log "has taken a dongle"

    4. acquire_dongle(dongle[second], c)
       → Essaie de prendre le SECOND dongle
       → Si echec → RELEASE le premier ! (TRES IMPORTANT) → return 1
       → Si OK → log "has taken a dongle"

    5. return 0 → les deux sont pris, on peut compiler

    SUPER IMPORTANT au point 4 : si l'acquisition du 2eme echoue, on DOIT
    release le 1er. Sinon ce 1er dongle reste marque comme "en utilisation"
    pour TOUJOURS, et tous les coders qui l'attendent sont bloques a jamais.
    C'est un cas de RESOURCE LEAK.

─── release_dongles() ───  ligne 61

    void    release_dongles(t_coder *c)

    Repose les 2 dongles en appelant release_dongle pour chacun.
    L'ordre de release n'a PAS d'importance : relacher une ressource
    ne peut jamais causer de deadlock (le deadlock c'est PRENDRE, pas LACHER).

C'est LE FICHIER LE PLUS COMPLEXE du projet. C'est ici que la concurrence,

Prends ton temps pour lire cette section. Relis-la 2 ou 3 fois si besoin.

─── make_request() ───  ligne 15

    static t_request    make_request(t_coder *c)

    Cree un "ticket" que le coder depose dans la file d'attente du dongle.
    C'est comme prendre un numero a la boulangerie.

    Le ticket contient :
    - req.coder_id = c->id → qui fait la demande
    - req.key → le numero de priorite (plus petit = plus prioritaire)

    Le calcul de la key depend du scheduler :

    Si FIFO :
        req.key = now_ms()
        → L'heure d'arrivee. Celui qui arrive en premier a la plus
        petite valeur (parce que le temps avance).
        → Premier arrive, premier servi. Simple.

    Si EDF (Earliest Deadline First) :
        req.key = get_last_compile(c) + sim->t_burnout
        → La deadline du coder = derniere compile + temps avant burnout
        → Celui qui va mourir le plus tot a la plus petite deadline
        → Plus urgent = servi en premier

    C'est ICI et UNIQUEMENT ICI que FIFO et EDF different. Tout le reste
    du code est identique : la priority queue trie par plus petite key,
    point. Elle ne sait pas ce que la key SIGNIFIE. C'est elegant.

─── can_take() ───  ligne 27

    static int  can_take(t_dongle *d, int coder_id)

    Verifie si le coder PEUT prendre le dongle MAINTENANT.
    TROIS conditions doivent etre TOUTES vraies :

    1. !d->in_use
       → Le dongle n'est pas utilise par quelqu'un d'autre.
       Si quelqu'un le tient → attendre.

    2. pq_peek(&d->waiters)->coder_id == coder_id
       → C'est MON TOUR dans la file d'attente.
       pq_peek regarde qui est en tete de la file. Si c'est moi → OK.
       Si c'est quelqu'un d'autre → attendre, meme si le dongle est libre !

    3. now_ms() >= d->released_at + d->cooldown
       → Le cooldown est fini.
       Le dongle a ete repose il y a assez longtemps.

    Si UNE SEULE condition est fausse → return 0 → le coder attend encore.

    La condition 2 est le SCHEDULER. C'est la que la "fairness" (equite)
    se joue. Meme si le dongle est libre, si un autre coder avec une
    meilleure priorite attend aussi, JE NE PASSE PAS. C'est la file
    d'attente qui decide, pas le hasard. C'est comme ca qu'on evite
    la STARVATION (un coder qui n'obtient jamais de dongle).

─── timed_wait() ───  ligne 41

    static void timed_wait(t_dongle *d)

    Fait dormir le thread en attendant un signal sur la sonnette du dongle.
    Avec un timeout de 10ms.

    Comment ca marche, etape par etape :

    1. gettimeofday(&tv, NULL) → recupere l'heure actuelle
       (secondes + microsecondes)

    2. On calcule "l'heure actuelle + 10ms" :
       - On ajoute 10000 microsecondes (= 10ms)
       - On convertit en nanosecondes (struct timespec)
       - Si les nanosecondes depassent 1 milliard (= 1 seconde),
         on ajuste : nanosecondes -= 1 milliard, secondes += 1
         C'est comme dire "3h 70min = 4h 10min"

    3. pthread_cond_timedwait(&d->cond, &d->lock, &timeout)
       Cette fonction fait 3 choses EN UNE SEULE OPERATION ATOMIQUE :
       a. LIBERE le mutex d->lock → pour que d'autres threads puissent
          modifier le dongle (release_dongle par exemple)
       b. DORT → le thread ne consomme plus de CPU
       c. Attend soit un SIGNAL (broadcast) soit le TIMEOUT (10ms)
       d. AU REVEIL → reprend le mutex d->lock automatiquement

    C'est crucial a comprendre : pendant que le coder dort, LE MUTEX EST
    LIBRE. Un autre coder peut donc appeler release_dongle, qui locke le
    mutex, met in_use=0, fait un broadcast (sonne la sonnette), et unlock.
    Le broadcast reveille notre coder endormi.

    Le timeout de 10ms est un filet de securite : meme si personne ne
    sonne (cas rare mais possible), le coder se reveille quand meme et
    re-verifie can_take() et sim_stopped(). Ca empeche de rester coince
    indefiniment.

─── acquire_dongle() ───  ligne 57  ★ FONCTION CRITIQUE ★

    int     acquire_dongle(t_dongle *d, t_coder *c)

    Acquiert UN dongle. C'est la fonction la plus importante du projet.
    Elle BLOQUE le thread jusqu'a ce que le coder obtienne le dongle
    ou que la simulation s'arrete.

    Flow detaille, comme un scenario :

    Imagine un coder qui arrive devant un dongle. Voila ce qu'il fait :

    1. make_request(c)
       → Il prend un ticket numerote

    2. pthread_mutex_lock(&d->lock)
       → Il entre dans la "zone protegee" du dongle (verrouille la porte)

    3. pq_push(&d->waiters, req)
       → Il depose son ticket dans la file d'attente du dongle.
       Le ticket est insere a la bonne position dans le heap (trie par key).

    4. WHILE (!can_take(d, c->id))
       → Il verifie : "c'est mon tour ET le dongle est libre ET cooldown ok ?"
       Si non :
       a. sim_stopped(c->sim) ?
          → Si la simulation est finie pendant l'attente :
          → Il retire son ticket de la file (pq_remove) — pour ne pas
            laisser un "fantome" qui bloquerait les autres
          → Il unlock et return 1 (echec)
       b. timed_wait(d)
          → Il s'endort (max 10ms, libere le mutex pendant le sommeil)
          → Au reveil → retour en haut du while → re-check can_take

    5. pq_pop(&d->waiters)
       → Son ticket est en tete → il le retire (c'est bien son tour)

    6. d->in_use = 1
       → Il marque le dongle comme "pris"

    7. pthread_mutex_unlock(&d->lock)
       → Il sort de la zone protegee

    8. return 0 (succes)

    POURQUOI pq_push AVANT le while et pq_pop APRES ?
    → Le coder doit etre DANS la file pour que can_take le voie en tete.
    → Il doit RESTER dans la file pendant qu'il attend.
    → Il se retire quand c'est effectivement son tour.

    POURQUOI pq_remove en cas de sim_stopped ?
    → Si le coder part sans se retirer de la pq, les autres coders
    verront un "fantome" en tete de file. Le fantome ne prendra jamais
    le dongle, mais personne d'autre ne peut passer car le fantome a la
    meilleure priorite. → BLOCAGE PERMANENT. C'est un bug subtil.

─── release_dongle() ───  ligne 80

    void    release_dongle(t_dongle *d)

    Repose le dongle sur la table. C'est l'inverse d'acquire.

    1. pthread_mutex_lock(&d->lock)
       → Entre dans la zone protegee

    2. d->in_use = 0
       → Le dongle est libre

    3. d->released_at = now_ms()
       → Note l'heure. Le cooldown demarre MAINTENANT.
       Le dongle ne sera reprenable que dans "cooldown" millisecondes.

    4. pthread_cond_broadcast(&d->cond)
       → Sonne la sonnette → reveille TOUS les coders endormis sur ce dongle

    5. pthread_mutex_unlock(&d->lock)
       → Sort de la zone protegee

    Pourquoi BROADCAST et pas SIGNAL ?
    - signal reveille UN SEUL thread. Mais lequel ? C'est aleatoire.
      Si le thread reveille n'est pas en tete de la pq, il se rendort,
      et le VRAI "prochain" dans la file reste endormi. Bug !
    - broadcast reveille TOUT LE MONDE. Chacun check can_take() :
      seul celui qui est en tete (ET dongle libre ET cooldown ok) passe.
      Les autres se rendorment immediatement.
    - C'est un tout petit peu plus de CPU (reveiller N threads au lieu
      de 1), mais c'est CORRECT. En concurrence, "correct" bat "rapide".


################################################################################
#                                                                              #
#   FICHIER 10 : srcs/priority_queue.c  (le min-heap)                         #
#                                                                              #
################################################################################

Un MIN-HEAP, c'est une structure de donnees qui fait UNE chose tres bien :
te donner le PLUS PETIT element instantanement.

Imagine une pile de dossiers ou le dossier le plus urgent est TOUJOURS
sur le dessus. Tu n'as pas besoin de fouiller, tu prends juste le premier.

Comment ca marche ? C'est un arbre binaire avec une regle simple :
chaque parent est PLUS PETIT que ses enfants.

        2          ← la racine est le minimum
       / \
      5   3
     / \
    8   7

On le stocke dans un TABLEAU PLAT (pas besoin de pointeurs left/right) :

    Index :  [0]  [1]  [2]  [3]  [4]
    Valeur:   2    5    3    8    7

    Pour naviguer dans l'arbre, on utilise des formules magiques :
    parent de i     = (i - 1) / 2       (division entiere)
    enfant gauche   = 2 * i + 1
    enfant droit    = 2 * i + 2

    Exemple : l'element a l'index 1 (valeur 5)
    - parent = (1-1)/2 = 0 → c'est l'element 2 ✓
    - gauche = 2*1+1 = 3 → c'est l'element 8 ✓
    - droite = 2*1+2 = 4 → c'est l'element 7 ✓

─── pq_init() ───  ligne 15

    int     pq_init(t_pq *pq, int cap, int (*cmp)(t_request, t_request))

    Cree une priority queue vide.
    - malloc un tableau de "cap" requests (cap = nombre de coders, parce
      qu'au maximum N coders peuvent attendre le meme dongle)
    - size = 0 (rien dedans au debut)
    - cmp = la fonction de comparaison (on lui passe cmp_request)

─── pq_push() ───  ligne 26

    void    pq_push(t_pq *pq, t_request req)

    Ajoute un ticket dans la file de priorite.

    Algorithme :
    1. Si la file est pleine → return (ne devrait jamais arriver en pratique)
    2. Place le nouveau ticket a la FIN du tableau (derniere position)
    3. size++
    4. sift_up → remonte le ticket vers le haut jusqu'a sa bonne place

    Imagine que tu mets une brique legere tout en bas d'une pile triee par
    poids. Elle est trop legere pour etre en bas, donc elle "remonte"
    en echangeant avec ses voisins plus lourds, jusqu'a trouver sa place.

    Complexite : O(log N) — au pire, le ticket remonte de la feuille
    jusqu'a la racine. Avec 100 coders, c'est ~7 echanges max.

─── pq_pop() ───  ligne 35

    t_request   pq_pop(t_pq *pq)

    Retire et retourne le ticket le plus prioritaire (celui a la racine).

    Algorithme :
    1. Sauvegarde data[0] (le ticket le plus prioritaire)
    2. size--
    3. Si il reste des tickets → met le DERNIER ticket a l'index 0
       (on "bouche le trou" en mettant le dernier element a la place du premier)
    4. sift_down → descend ce ticket jusqu'a sa bonne place
       (il est probablement trop "lourd" pour etre en haut)
    5. Retourne l'ancien ticket minimum

    Complexite : O(log N)

─── pq_peek() ───  ligne 49

    t_request   *pq_peek(t_pq *pq)

    Retourne un POINTEUR vers le ticket en tete, SANS le retirer.
    Si la file est vide → retourne NULL.

    Utilisee par can_take() pour verifier "est-ce que JE suis en tete
    de la file ?" sans modifier la file.

─── pq_destroy() ───  ligne 56

    void    pq_destroy(t_pq *pq)

    Free le tableau et remet tout a zero. Appele pendant destroy_sim.

#                                                                              #
#   FICHIER 11 : srcs/pq_helpers.c  (les entrailles du heap)                  #
#                                                                              #
################################################################################

Les fonctions internes qui maintiennent la propriete du heap
(parent <= enfants) quand on ajoute ou retire des elements.

─── pq_swap() ───  ligne 15

    static void pq_swap(t_pq *pq, int a, int b)

    Echange deux elements dans le tableau. data[a] ↔ data[b].
    Classique, rien de special.

─── sift_up() ───  ligne 24

    void    sift_up(t_pq *pq, int i)

    Remonte l'element a l'index i vers la racine (le haut de l'arbre).
    Utilisee apres un pq_push (le nouvel element est en bas et doit
    remonter a sa place).

    Algorithme en langage humain :
    "Tant que je ne suis pas a la racine :
     - Je regarde mon parent
     - Si mon parent est plus gros que moi → on echange de place
     - Je continue de remonter
     - Si mon parent est plus petit ou egal → je suis a ma place, stop"

    En code :
    while (i > 0)                           → tant qu'on n'est pas a la racine
        parent = (i - 1) / 2               → index du parent
        if (!cmp(data[parent], data[i]))    → si parent est bien place, stop
            break
        pq_swap(pq, i, parent)              → sinon, on echange
        i = parent                          → et on continue vers le haut

    Comment lire la fonction cmp :
    cmp(a, b) retourne 1 si "a devrait etre EN DESSOUS de b dans le heap"
    (= a est moins prioritaire que b).
    Donc cmp(parent, child) = 1 → "parent est trop gros, swap !"
    Et cmp(parent, child) = 0 → "parent est bien place, stop."

─── sift_down() ───  ligne 38

    void    sift_down(t_pq *pq, int i)

    Descend l'element a l'index i vers les feuilles (le bas de l'arbre).
    Utilisee apres un pq_pop (on a mis le dernier element a la racine
    et il doit descendre a sa place).

    Algorithme en langage humain :
    "En boucle :
     - Je regarde mes deux enfants (gauche et droit)
     - Je trouve le plus petit des trois (moi + gauche + droit)
     - Si c'est moi le plus petit → je suis a ma place, stop
     - Sinon → j'echange avec le plus petit enfant
     - Je continue de descendre"

    En code :
    while (1)
        min = i                        → suppose que je suis le min
        l = 2*i + 1, r = 2*i + 2      → mes enfants gauche et droit
        si l existe ET data[min] > data[l] → min = l  (gauche est plus petit)
        si r existe ET data[min] > data[r] → min = r  (droit est encore plus petit)
        si min == i → je suis bien place, break
        sinon → swap(i, min), i = min, continuer

    On compare AVEC LES DEUX ENFANTS et on swap avec le plus petit.
    Ca garantit que la propriete min-heap (parent <= enfants) est restauree.

─── cmp_request() ───  ligne 62

    int     cmp_request(t_request a, t_request b)

    Retourne 1 si a devrait etre APRES b (= a est moins prioritaire).

    if (a.key == b.key)                → en cas d'egalite de priorite
        return (a.coder_id > b.coder_id)  → plus petit id passe d'abord
    return (a.key > b.key)             → plus petite key = plus prioritaire

    C'est une regle simple : compare les keys. En cas d'egalite, le coder
    avec le plus petit ID passe d'abord (arbitrage deterministe).

    Cette fonction est la MEME pour FIFO et EDF. La difference entre les
    deux schedulers est dans le CALCUL de la key (fait dans make_request),
    pas dans la comparaison. La pq ne sait pas ce que la key represente.

─── pq_remove() ───  ligne 69

    void    pq_remove(t_pq *pq, int coder_id)

    Retire un coder specifique de la file (utilisee quand la simulation
    s'arrete et qu'un coder doit partir sans prendre le dongle).

    Algorithme :
    1. Parcourt le tableau lineairement pour trouver le coder (O(N))
    2. Remplace l'element trouve par le DERNIER element du tableau
    3. size--
    4. Restaure la propriete du heap avec sift_down PUIS sift_up
       → On ne sait pas si le remplacement est trop grand ou trop petit
       → sift_down gere le cas "trop grand", sift_up gere "trop petit"
       → L'un des deux sera un no-op, et c'est OK

#                                                                              #
#   FICHIER 12 : srcs/monitor.c  (le surveillant)                             #
#                                                                              #
################################################################################

Le monitor est un thread special qui ne compile pas, ne prend pas de dongle.
Il fait UNE chose : surveiller. C'est le professeur qui patrouille dans la
salle d'examen pour verifier que personne ne triche... ou ne s'endort.

─── check_all_done() ───  ligne 15

    static int  check_all_done(t_sim *sim)

    Parcourt TOUS les coders. Pour chacun, lit compiles_done (sous mutex)
    et verifie s'il a atteint compiles_req.

    Si TOUS les coders ont fini → retourne 1 (succes ! simulation terminee)
    Si au moins un n'a pas fini → retourne 0 (on continue)

    Note : chaque lecture de compiles_done passe par get_compiles_done()
    qui locke le mutex du coder. C'est thread-safe.

─── check_burnouts() ───  ligne 28

    static int  check_burnouts(t_sim *sim)

    Parcourt TOUS les coders. Pour chacun :
    1. Lit last_compile (sous mutex) via get_last_compile()
    2. Calcule : since = now_ms() - last_compile
       → "Ca fait combien de temps qu'il n'a pas compile ?"
    3. Si since >= t_burnout → BURNOUT ! Ce coder est mort.

    Retourne l'ID du PREMIER coder en burnout (1..N), ou 0 si personne.

    Pourquoi seulement le premier ? Parce qu'une fois qu'on detecte un
    burnout, on arrete tout. Les autres burnouts eventuels n'ont pas
    d'importance.

─── log_burnout() ───  ligne 43

    static void log_burnout(t_sim *sim, int id)

    Affiche "X burned out" et arrete la simulation.

    1. Lock log_mtx
    2. if (!sim->stop) → verifie qu'on n'a pas DEJA arrete
       (si deux burnouts arrivent au meme instant, on ne logge qu'une fois)
    3. printf("%ld %d burned out\n", timestamp, id)
    4. sim->stop = 1
    5. Unlock

    Le check "if (!stop)" sous le lock est crucial :
    - Sans lui, si 2 coders burnoutent en meme temps, on verrait 2 messages
      "burned out" et stop serait set 2 fois (inoffensif mais sale)
    - Le printf ET le set de stop sont sous LE MEME LOCK. Ca garantit
      qu'aucun autre log ne passe entre "burned out" et stop=1.

─── monitor_routine() ───  ligne 55  ★ LE GARDIEN ★

    void    *monitor_routine(void *arg)

    C'est la routine du thread monitor. Boucle infinie :

    while (1)
    {
        1. check_all_done(sim)
           → Si tous les coders ont fini : set stop = 1, return (fin propre)

        2. check_burnouts(sim)
           → Si un coder a fait burnout : log_burnout, return (fin avec mort)

        3. usleep(500)
           → Dort 0.5 milliseconde, puis recommence le check
    }

    Pourquoi 500 microsecondes (0.5ms) ?
    C'est un compromis entre precision et performance :
    - Trop court (100us) → detecte plus vite, mais bouffe du CPU inutilement
    - Trop long (5000us = 5ms) → economise le CPU, mais peut detecter un
      burnout avec 5ms de retard (probleme pour les timings serres)
    - 500us → ~2000 checks par seconde, precision < 1ms, c'est bien

    L'ORDRE des checks est DELIBERE :
    → check_all_done est teste AVANT check_burnouts
    → Pourquoi ? Si tous les coders ont fini ET qu'un coder est pile
      sur sa deadline de burnout, on considere que c'est un SUCCES,
      pas un echec. Le doute beneficie au coder.

    Le "return (NULL)" final apres le while(1) ne s'executera JAMAIS
    (la boucle est infinie, on sort via les return internes). Mais sans
    lui, le compilateur pourrait avertir "control reaches end of non-void
    function". Donc on le garde pour etre propre.

#   FICHIER 13 : Makefile                                                      #

################################################################################

Le Makefile, c'est la "recette de construction" du projet. Au lieu de taper
la commande de compilation a la main a chaque fois, on ecrit les regles
une fois et on tape juste "make".

    NAME    = codexion          → le nom du binaire final (l'executable)
    CC      = cc                → le compilateur (generalement gcc ou clang)
    CFLAGS  = -Wall -Wextra -Werror -pthread

    Explication des flags de compilation :
    -Wall    → active la plupart des warnings (avertissements)
               "Hey, cette variable n'est pas utilisee" etc.
    -Wextra  → active des warnings supplementaires (encore plus strict)
    -Werror  → traite les warnings comme des ERREURS
               → Le code ne compile pas s'il y a un seul warning
               → Ca force a ecrire du code propre
    -pthread → active le support des threads POSIX (pthread_create etc.)
               Sans ce flag, le linker ne trouvera pas les fonctions pthread.

    SRCS = srcs/main.c \        → liste de tous les fichiers source
           srcs/parse_args.c \     Le "\" en fin de ligne veut dire
           ...                      "la ligne continue en dessous"

    OBJS = $(SRCS:.c=.o)       → magie du Makefile : remplace .c par .o
                                  srcs/main.c devient srcs/main.o
                                  Ce sont les fichiers "compiles mais pas lies"

    all: $(NAME)               → la cible par defaut.
                                  "make" ou "make all" construit le binaire.

    $(NAME): $(OBJS)           → le binaire depend de tous les .o
        $(CC) $(CFLAGS) $(OBJS) -o $(NAME)
        → Lie tous les .o ensemble pour creer l'executable final.
        "-o codexion" = "le fichier de sortie s'appelle codexion"

    %.o: %.c includes/codexion.h   → regle generique : pour chaque .c,
                                     cree un .o
        $(CC) $(CFLAGS) -I includes -c $< -o $@
        -I includes = "cherche les headers dans le dossier includes/"
        -c = "compile seulement, ne lie pas encore"
        $< = le fichier source (.c) — variable automatique de make
        $@ = le fichier cible (.o) — variable automatique de make

    Si codexion.h change, TOUT est recompile (parce qu'il est en dependance).

    clean:    → "make clean" → supprime les fichiers .o (les intermediaires)
    fclean:   → "make fclean" → supprime les .o ET le binaire final
    re:       → "make re" → fclean + all (recompile tout de zero)

    .PHONY: all clean fclean re
    → Dit a make : "all, clean, fclean, re ne sont PAS des fichiers."
    Sans ca, si tu as un fichier nomme "clean" dans ton dossier,
    make dirait "clean is up to date" et ne ferait rien. .PHONY evite ca.

    COMPILATION INCREMENTALE : make est intelligent. Il regarde les
    timestamps (dates de modification) des fichiers. Si tu modifies
    SEULEMENT coder.c, make ne recompile QUE coder.o et relie le binaire.
    Les autres .o ne sont pas retouches. C'est beaucoup plus rapide que
    de tout recompiler a chaque fois.

#   COMMENT TOUT S'ASSEMBLE (le flow complet de A a Z)                        #
#                                                                              #
################################################################################

Voici ce qui se passe quand tu lances :

    ./codexion 5 800 200 200 200 3 50 fifo

    Traduction : 5 coders, burnout a 800ms, compile 200ms, debug 200ms,
    refactor 200ms, 3 compiles chacun, cooldown 50ms, scheduler FIFO.

    PHASE 1 — PREPARATION (un seul thread, le main)

    1. main() appelle parse_args()
       → Lit "5 800 200 200 200 3 50 fifo"
       → Remplit la struct sim avec ces valeurs
       → Si erreur → "Error\n" et on quitte

    2. init_sim()
       → malloc 5 dongles (dongle 0 a 4)
       → Pour chaque dongle : init mutex + cond + priority queue
       → malloc 5 coders (coder 1 a 5)
       → Pour chaque coder : init mutex
       → Init le log_mtx

    3. set_start_time()
       → sim->start_ms = heure actuelle (ex: 1000000)
       → Pour chaque coder : last_compile = 1000000
       → Le chronometre demarre

    4. launch_threads()
       → Cree 5 threads coder (executant coder_routine)
       → Cree 1 thread monitor (executant monitor_routine)

    === A PARTIR D'ICI, 7 THREADS TOURNENT EN PARALLELE ===
    (5 coders + 1 monitor + le main qui attend)

    PHASE 2 — EXECUTION (6 threads actifs en parallele)

    CHAQUE CODER fait ca en boucle :
    ┌──────────────────────────────────────────────────────┐
    │ while (!stop)                                        │
    │                                                      │
    │   ① Prendre 2 dongles (peut bloquer longtemps)       │
    │      → acquire premier dongle :                      │
    │        lock → push ticket → while(!can_take) {       │
    │          sim_stopped? → timed_wait 10ms } →          │
    │        pop ticket → in_use=1 → unlock                │
    │      → acquire deuxieme dongle (pareil)              │
    │      → log "has taken a dongle" (x2)                 │
    │                                                      │
    │   ② Compiler (200ms)                                 │
    │      → update_last_compile (reset chrono burnout)    │
    │      → log "is compiling"                            │
    │      → precise_sleep(200ms) (micro-siestes de 0.2ms) │
    │      → release les 2 dongles (in_use=0 + broadcast)  │
    │      → inc_compiles_done (+1)                        │
    │                                                      │
    │   ③ Debugger (200ms, sans dongle)                    │
    │      → log "is debugging"                            │
    │      → precise_sleep(200ms)                          │
    │                                                      │
    │   ④ Refactorer (200ms, sans dongle)                  │
    │      → log "is refactoring"                          │
    │      → precise_sleep(200ms)                          │
    │                                                      │
    │   → Retour a ①                                       │
    └──────────────────────────────────────────────────────┘

    LE MONITOR fait ca en boucle :
    ┌──────────────────────────────────────────────────────┐
    │ while (1)                                            │
    │   Tous ont fini ? → lock chaque c->mtx, lit compiles│
    │     Si oui → stop=1, return (SUCCES)                │
    │   Quelqu'un en burnout ?                            │
    │     → lock chaque c->mtx, lit last_compile          │
    │     → now - last_compile >= 800 ? → MORT            │
    │     Si oui → log "burned out", stop=1, return       │
    │   usleep(500) → dort 0.5ms, recommence              │
    └──────────────────────────────────────────────────────┘

    PHASE 3 — ARRET (quand stop passe a 1)

    → Les coders dans precise_sleep sortent en < 0.2ms
      (ils checkent sim_stopped a chaque micro-sieste)
    → Les coders dans timed_wait sortent en < 10ms
      (le timeout expire et ils voient que sim_stopped est vrai)
    → Chaque thread fait "return NULL" = il meurt

    PHASE 4 — NETTOYAGE (retour au main thread seul)

    → join_threads() attend que les 6 threads finissent
    → destroy_sim() free tout : dongles, coders, mutex, cond, pq
    → return 0 → le programme quitte proprement

#                                                                              #
#   LES MUTEX ET CE QU'ILS PROTEGENT (resume)                                 #
#                                                                              #
################################################################################

    MUTEX               PROTEGE                         QUI L'UTILISE
    ──────────────────────────────────────────────────────────────────────
    sim->log_mtx        sim->stop (le flag d'arret)     Tous les threads
                        stdout (printf)                 Tous les threads

    c->mtx              c->last_compile                 Coder + Monitor
    (un par coder)      c->compiles_done                Coder + Monitor

    d->lock             d->in_use                       Les Coders
    (un par dongle)     d->released_at                  Les Coders
                        d->waiters (la pq)              Les Coders
                        d->cond (la sonnette)           Les Coders

    REGLES pour eviter le deadlock entre mutex differents :

    1. On ne prend JAMAIS deux mutex "de meme type" en meme temps
       SAUF pour les dongles → mais l'ordre des IDs empeche le cycle

    2. L'ordre de prise est toujours :
       d->lock (dongle) avant c->mtx (coder) avant sim->log_mtx (log)
       → Pas d'inversion possible → pas de deadlock entre types de mutex


################################################################################
#                                                                              #
#   POURQUOI IL N'Y A PAS DE DEADLOCK (la preuve)                             #
#                                                                              #
################################################################################

    Un deadlock necessite 4 conditions (les conditions de Coffman).
    Si on en casse UNE SEULE, le deadlock est IMPOSSIBLE.

    On casse le CIRCULAR WAIT (attente circulaire).

    Regle : un coder prend TOUJOURS le dongle de plus petit ID d'abord.

    Preuve par contradiction (raisonnement mathematique) :

    Supposons qu'un deadlock existe. Ca veut dire qu'il y a un CYCLE :
    - Coder A tient dongle D1 et attend dongle D2 (tenu par B)
    - Coder B tient dongle D2 et attend dongle D3 (tenu par C)
    - Coder C tient dongle D3 et attend dongle D1 (tenu par A)

    Mais notre regle dit : on prend le plus petit ID d'abord.
    Donc si A tient D1 et attend D2, c'est que D1 < D2 (il a pris le petit d'abord)
    De meme : B tient D2 et attend D3 → D2 < D3
    De meme : C tient D3 et attend D1 → D3 < D1

    Conclusion : D1 < D2 < D3 < D1

    C'est IMPOSSIBLE. Un nombre ne peut pas etre plus petit que lui-meme.
    Contradiction. Donc notre hypothese est fausse.
    Donc le cycle n'existe pas. Donc il n'y a PAS de deadlock. CQFD.

    THREADS ET SYNCHRONISATION :

    pthread_cond_t      → le TYPE d'une sonnette (condition variable)
    pthread_cond_init   → initialise une sonnette
    pthread_cond_wait   → s'endort en attendant que quelqu'un sonne
                          (libere le cadenas pendant le sommeil)
    pthread_cond_timedwait → pareil mais avec un reveil automatique
    pthread_cond_broadcast → sonne pour reveiller TOUS les dormeurs
    pthread_cond_signal → sonne pour reveiller UN SEUL dormeur (aleatoire)
    pthread_cond_destroy → detruire la sonnette

    TEMPS :
    gettimeofday        → donne l'heure actuelle (secondes + microsecondes)
    usleep              → dort N microsecondes (1ms = 1000 microsecondes)

    MEMOIRE :
    malloc              → reserve un bloc de memoire (demander un terrain)
    free                → libere un bloc de memoire (rendre le terrain)
    memset              → remplit un bloc avec une valeur (tout mettre a zero)

    STRINGS :
    strcmp              → compare 2 strings (retourne 0 si identiques)

    AFFICHAGE :
    printf              → affiche du texte formate sur la sortie standard
