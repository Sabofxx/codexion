─── LIGNES 13-14 : Include guard ───

    #ifndef CODEXION_H
    # define CODEXION_H

    Empeche d'inclure le header 2 fois. Si codexion.h est deja inclus,
    CODEXION_H est deja defini, donc tout le fichier est saute.
    Sans ca, le compilateur crierait "type deja defini".

─── LIGNES 16-21 : Includes systeme ───

    # include <pthread.h>       → threads, mutex, cond vars
    # include <sys/time.h>      → gettimeofday (pour mesurer le temps)
    # include <unistd.h>        → usleep, write
    # include <stdio.h>         → printf
    # include <stdlib.h>        → malloc, free
    # include <string.h>        → strcmp, memset

    Ce sont les SEULES librairies autorisees par le sujet.

─── LIGNES 23-27 : Enum t_sched ───

    typedef enum e_sched
    {
        SCH_FIFO,     → vaut 0
        SCH_EDF       → vaut 1
    }   t_sched;

    Un enum c'est juste des noms pour des nombres.
    Au lieu d'ecrire "if (scheduler == 0)", on ecrit "if (scheduler == SCH_FIFO)".
    Plus lisible.

    Note : on a du renommer de "SCHED_FIFO" a "SCH_FIFO" parce que macOS
    definit deja SCHED_FIFO dans ses headers systeme. Conflit de noms.

─── LIGNES 29-33 : Struct t_request ───

    typedef struct s_request
    {
        int     coder_id;   → quel coder a fait cette demande (1..N)
        long    key;        → cle de priorite
    }   t_request;

    C'est un "ticket" qu'un coder depose dans la file d'attente d'un dongle.

    La "key" depend du scheduler :
    - FIFO : key = timestamp d'arrivee (premier arrive = plus petite key)
    - EDF  : key = deadline du coder = last_compile + time_to_burnout
             (celui qui va mourir le plus tot = plus petite key)

    Dans les deux cas : plus petite key = plus haute priorite.

─── LIGNES 35-41 : Struct t_pq (Priority Queue) ───

    typedef struct s_pq
    {
        t_request   *data;   → tableau dynamique de requests
        int         size;    → nombre d'elements actuels
        int         cap;     → capacite max (= nombre de coders)
        int         (*cmp)(t_request, t_request);  → fonction de comparaison
    }   t_pq;

    C'est notre MIN-HEAP (tas binaire). Stocke en tableau plat.

    Le champ (*cmp) est un POINTEUR DE FONCTION. C'est comme une variable
    qui contient une fonction. On y met "cmp_request" a l'init.
    Ca permet a la pq de comparer deux requests sans savoir comment.

    Pourquoi un heap et pas une liste triee ?
    → push et pop en O(log N) au lieu de O(N). Le sujet l'impose.

─── LIGNE 43 : Forward declaration ───

    typedef struct s_sim    t_sim;

    On dit au compilateur "t_sim existe, je te donne les details plus tard".
    C'est necessaire parce que t_dongle et t_coder ont des pointeurs vers
    t_sim, mais t_sim contient des pointeurs vers t_dongle et t_coder.
    Reference circulaire → il faut declarer un des deux en avance.

─── LIGNES 45-55 : Struct t_dongle ───

    typedef struct s_dongle
    {
        int             id;           → numero du dongle (0..N-1)
        int             in_use;       → 1 si un coder le tient, 0 sinon
        int             inited;       → 1 si mutex/cond/pq initialises
        long            released_at;  → timestamp ms de la derniere release
        long            cooldown;     → duree du cooldown en ms
        pthread_mutex_t lock;         → cadenas pour proteger cette struct
        pthread_cond_t  cond;         → "sonnette" pour reveiller les waiters
        t_pq            waiters;      → file d'attente des coders
    }   t_dongle;

    Chaque dongle est une RESSOURCE PARTAGEE. Plusieurs coders la veulent.

    Le "lock" (mutex) protege TOUT dans cette struct. Avant de lire ou
    modifier in_use, released_at, ou waiters, tu dois lock.

    Le "cond" sert a dire "un truc a change, reveillez-vous".
    Quand un coder release le dongle, il fait broadcast sur cond.
    Les coders endormis dans timed_wait se reveillent et re-checkent.

    "released_at" + "cooldown" = quand le dongle redevient prenable.
    Si now_ms() < released_at + cooldown, le dongle est encore chaud.

    "inited" sert au cleanup : on ne detruit que ce qui a ete initialise.

─── LIGNES 57-66 : Struct t_coder ───

    typedef struct s_coder
    {
        int             id;             → numero affiche (1..N)
        int             inited;         → 1 si mutex initialisee
        pthread_t       thread;         → identifiant du thread pthread
        int             compiles_done;  → combien de compiles termines
        long            last_compile;   → timestamp ms du dernier compile
        pthread_mutex_t mtx;            → protege compiles_done et last_compile
        t_sim           *sim;           → pointeur vers la struct mere
    }   t_coder;

    Chaque coder = 1 thread. Le thread recoit un pointeur vers sa struct
    t_coder en argument. Via c->sim, il accede a tout (dongles, config...).

    "mtx" protege les deux champs qui sont lus par le monitor ET ecrits
    par le coder en parallele : compiles_done et last_compile.
    Sans ce mutex → race condition → valeurs corrompues → burnout faux.

    "id" est 1-indexe pour l'affichage ("1 is compiling", pas "0 is compiling").
    En interne, le coder est stocke dans sim->coders[id - 1].

─── LIGNES 68-86 : Struct s_sim (la structure mere) ───

    struct s_sim
    {
        int             n;              → nombre de coders (et de dongles)
        long            t_burnout;      → ms avant burnout
        long            t_compile;      → ms pour compiler
        long            t_debug;        → ms pour debugger
        long            t_refactor;     → ms pour refactorer
        int             compiles_req;   → nb de compiles requis pour finir
        long            cooldown;       → ms de cooldown par dongle
        t_sched         scheduler;      → SCH_FIFO ou SCH_EDF
        long            start_ms;       → timestamp ms du debut de la simu
        int             stop;           → flag d'arret (0 = on tourne, 1 = stop)
        int             n_threads;      → nb de threads coder crees avec succes
        int             mon_created;    → 1 si le thread monitor a ete cree
        pthread_mutex_t log_mtx;        → protege stop ET serialise les logs
        t_dongle        *dongles;       → tableau de N dongles
        t_coder         *coders;        → tableau de N coders
        pthread_t       monitor;        → thread du surveillant
    };

    C'est LE CENTRE NEVRALGIQUE. Tout passe par cette struct.
    Elle est creee sur la pile (stack) dans main(), et un pointeur est
    passe a chaque thread via c->sim ou directement en argument.

    PAS DE VARIABLE GLOBALE : tout est dans t_sim.

    "log_mtx" fait DOUBLE emploi :
    1. Il protege "stop" (le flag d'arret)
    2. Il serialise les printf (pas de melange de lignes)

    Pourquoi un seul mutex pour les deux ? Parce qu'on veut que le check
    de "stop" et l'ecriture du log soient ATOMIQUES. Si on checkait stop
    avec un mutex et loggait avec un autre, un log pourrait passer APRES
    "burned out". Avec un seul mutex, c'est impossible.

─── now_ms() ───  ligne 15

    long    now_ms(void)
    {
        struct timeval  tv;

        gettimeofday(&tv, NULL);
        return (tv.tv_sec * 1000L + tv.tv_usec / 1000);
    }

    Retourne le temps actuel en MILLISECONDES depuis le 1er janvier 1970.

    gettimeofday remplit une struct avec :
    - tv_sec  = secondes (genre 1713000000)
    - tv_usec = microsecondes (0 a 999999)

    On convertit : sec * 1000 + usec / 1000 = millisecondes.

    Le "1000L" (avec un L) force le calcul en long. Sans ca, sur certaines
    machines, tv_sec * 1000 pourrait depasser la taille d'un int et wraparound.

    Cette fonction est appelee des DIZAINES de fois par seconde par chaque thread.

─── precise_sleep() ───  ligne 23

    void    precise_sleep(long ms, t_sim *sim)
    {
        long    start;

        start = now_ms();
        while (now_ms() - start < ms)
        {
            if (sim_stopped(sim))
                return ;
            usleep(200);
        }
    }

    Fait dormir le thread pendant "ms" millisecondes, MAIS :
    1. Sort immediatement si la simulation s'arrete (sim_stopped)
    2. Est PRECISE (a ~0.2ms pres)

    Pourquoi pas juste usleep(ms * 1000) ?
    → usleep n'est PAS precis. L'OS peut te reveiller 5-20ms plus tard.
    → Et si la simu s'arrete pendant le sleep, tu veux sortir TOUT DE SUITE,
      pas attendre la fin du sleep.

    L'astuce : on fait plein de MICRO-SLEEPS de 200 microsecondes (0.2ms),
    et entre chaque, on re-verifie l'heure et le flag stop.

    Le cout : un peu de CPU (on se reveille 5000 fois par seconde). Mais c'est
    negligeable et ca donne une precision de 0.2ms — bien meilleur que les
    10ms demandees par le sujet.

─── log_state() ───  ligne 36

    void    log_state(t_coder *c, const char *msg)
    {
        pthread_mutex_lock(&c->sim->log_mtx);
        if (!c->sim->stop)
            printf("%ld %d %s\n", now_ms() - c->sim->start_ms, c->id, msg);
        pthread_mutex_unlock(&c->sim->log_mtx);
    }

    Affiche un message de log, format : "timestamp_ms id message\n"

    POINT CRITIQUE : tout est sous log_mtx.

    Pourquoi ? Deux raisons :
    1. SERIALISATION : si deux threads printf en meme temps, les caracteres
       se melangent. Le mutex garantit qu'un seul thread ecrit a la fois.
    2. COHERENCE AVEC STOP : on check "stop" SOUS LE LOCK. Si stop vaut 1,
       on n'ecrit rien. Ca garantit qu'aucun log ne sort APRES "burned out".

    "now_ms() - start_ms" donne le temps relatif depuis le debut de la simu.
    C'est ce que le sujet demande (pas le temps absolu).

─── sim_stopped() ───  ligne 44

    int     sim_stopped(t_sim *sim)
    {
        int     val;

        pthread_mutex_lock(&sim->log_mtx);
        val = sim->stop;
        pthread_mutex_unlock(&sim->log_mtx);
        return (val);
    }

    Lit le flag "stop" de maniere THREAD-SAFE.

    Sans le mutex, un thread pourrait lire "stop" en meme temps qu'un autre
    l'ecrit. Sur la plupart des CPUs, lire un int est atomique, mais le
    standard C ne le garantit pas. Et surtout, sans mutex, le compilateur
    pourrait CACHER la valeur dans un registre et ne jamais re-lire la memoire
    (optimisation). Le mutex force une "barriere memoire" qui dit au CPU
    "re-lis la vraie valeur de la RAM".

─── parse_one() ───  ligne 15

    static int  parse_one(char *str, long *out)

    Convertit une chaine de caracteres en nombre long.
    Retourne 0 si OK, 1 si erreur.

    Ligne par ligne :
    - val = 0; i = 0;       → initialise le resultat et l'index
    - if (!str[0])           → chaine vide ? erreur
    - while (str[i])         → parcourt chaque caractere
    - if (str[i] < '0' || str[i] > '9')  → pas un chiffre ? erreur
    - val = val * 10 + (str[i] - '0')    → construit le nombre
      Exemple : "123" → val=1, val=12, val=123
    - *out = val             → ecrit le resultat dans le pointeur
    - return (0)             → succes

    Pourquoi pas atoi() ? Parce que atoi("abc") retourne 0 sans erreur.
    On veut REJETER "abc", "-5", "" etc. D'ou le parsing manuel.

─── parse_numbers() ───  ligne 35

    static int  parse_numbers(t_sim *sim, char **av)

    Parse les 7 arguments numeriques (av[1] a av[7]) et les stocke
    dans la struct sim.

    av[1] = number_of_coders     → doit etre >= 1 (d'ou le "|| tmp < 1")
    av[2] = time_to_burnout      → long, >= 0
    av[3] = time_to_compile      → long, >= 0
    av[4] = time_to_debug        → long, >= 0
    av[5] = time_to_refactor     → long, >= 0
    av[6] = compiles_required    → int, >= 0
    av[7] = dongle_cooldown      → long, >= 0

    Pour n et compiles_req, on parse dans un "tmp" long d'abord, puis on
    caste en int. C'est parce que parse_one ecrit un long*, et ces champs
    sont des int dans la struct.

─── init_one_dongle() ───  ligne 15

    static int  init_one_dongle(t_dongle *d, int id, t_sim *sim)

    Initialise UN dongle. Ordre critique :
    1. Set les valeurs simples (id, in_use=0, released_at=0, cooldown)
    2. inited = 0 (pas encore pret)
    3. pthread_mutex_init → si ca plante, return 1
    4. pthread_cond_init → si ca plante, detruire le mutex deja cree
    5. pq_init → si ca plante, detruire cond ET mutex
    6. inited = 1 (tout est bon)

    C'est un pattern classique : "cleanup on error". Si l'etape 4 plante,
    tu dois defaire l'etape 3 avant de return. Sinon = fuite de ressource.

    pq_init recoit "cmp_request" comme fonction de comparaison. C'est la
    meme pour FIFO et EDF — la difference est dans la KEY, pas dans le cmp.

─── init_dongles() ───  ligne 39

    static int  init_dongles(t_sim *sim)

    1. malloc un tableau de N dongles
    2. memset a 0 (tous les champs a zero, y compris "inited")
    3. init chaque dongle un par un

    Le memset est crucial : si init_one_dongle echoue au dongle 3,
    les dongles 4 et 5 auront inited=0 grace au memset. Le cleanup
    saura qu'il ne doit pas les detruire.

─── init_one_coder() ───  ligne 56

    static int  init_one_coder(t_coder *c, int id, t_sim *sim)

    - c->id = id + 1    → id interne est 0-indexe, affichage est 1-indexe
    - c->last_compile = 0 → temporaire, sera mis a start_ms plus tard
    - c->sim = sim       → back-pointer vers la struct mere
    - pthread_mutex_init → le mutex personnel du coder

─── init_coders() ───  ligne 69

    Meme pattern que init_dongles. malloc + memset + boucle d'init.

─── init_sim() ───  ligne 86

    int     init_sim(t_sim *sim)

    Le chef d'orchestre de l'initialisation.
    1. stop = 0, pointeurs = NULL, compteurs = 0
    2. Init log_mtx (le mutex global)
    3. Init dongles ET coders
    4. Si un des deux echoue → destroy_sim (qui ne detruit que ce qui existe)

    Mettre les pointeurs a NULL AVANT d'appeler destroy_sim est essentiel.
    destroy_sim checke "if (sim->dongles)" avant de free. Si dongles etait
    un pointeur poubelle (non-NULL mais non-malloc), free() crasherait.

─── destroy_one_dongle() ───  ligne 15

    static void destroy_one_dongle(t_dongle *d)

    Detruit les 3 ressources d'un dongle dans l'ORDRE INVERSE de creation :
    1. pq_destroy (free le tableau de la pq)
    2. pthread_cond_destroy
    3. pthread_mutex_destroy

    Ordre inverse = bonne pratique. On detruit d'abord ce qui a ete cree en
    dernier. C'est comme depiler : LIFO (Last In First Out).

─── destroy_sim() ───  ligne 22

    void    destroy_sim(t_sim *sim)

    Nettoie TOUT :
    1. Si dongles existe → boucle, detruit chaque dongle INITIALISE, free tableau
    2. Si coders existe → boucle, detruit chaque mutex INITIALISE, free tableau
    3. Detruit log_mtx

    Le check "if (sim->dongles[i].inited)" evite de detruire un mutex non
    initialise, ce qui serait un comportement indefini (crash possible).

    Cette fonction est appelee :
    - En cas d'erreur pendant init_sim (cleanup partiel)
    - A la fin de main (cleanup normal apres simulation)

─── set_start_time() ───  ligne 15

    static void set_start_time(t_sim *sim)

    Appele JUSTE AVANT le lancement des threads.
    Fait deux choses :
    1. sim->start_ms = now_ms() → note le temps de reference (T0)
    2. Pour chaque coder, last_compile = start_ms

    Pourquoi on fait ca ICI et pas dans init_sim ?
    → Precision. Entre init_sim et launch_threads, il y a un delai (malloc,
    mutex_init...). Si on notait start_ms dans init_sim, les coders auraient
    deja "perdu" quelques ms de leur timer burnout avant meme de demarrer.
    En le faisant juste avant les threads, le chrono est aussi precis que possible.

    Pas besoin de mutex ici : aucun thread n'est encore lance.

─── launch_threads() ───  ligne 25

    static int  launch_threads(t_sim *sim)

    Cree N threads coder + 1 thread monitor.

    - n_threads = 0 → compteur de threads crees avec succes
    - Boucle : pthread_create pour chaque coder
      - En cas d'echec → stop = 1, return 1
      - Sinon → n_threads++
    - pthread_create pour le monitor
    - mon_created = 1

    pthread_create prend 4 arguments :
    1. &thread → ou stocker l'identifiant du thread
    2. NULL → attributs par defaut
    3. coder_routine → la FONCTION que le thread va executer
    4. &sim->coders[i] → l'ARGUMENT passe a cette fonction (void*)

    Si la creation echoue (memoire insuffisante, trop de threads...) :
    - stop = 1 → les threads deja crees verront le flag et sortiront
    - return 1 → main() saura que ca a foire

    n_threads et mon_created permettent a join_threads de ne joindre que
    les threads qui existent vraiment.

─── join_threads() ───  ligne 49

    static void join_threads(t_sim *sim)

    pthread_join = "attends que ce thread finisse".

    On join chaque thread coder (0 a n_threads-1) puis le monitor.
    Sans join, main() pourrait return avant que les threads aient fini →
    les threads sont tues brutalement → fuites, crash, logs tronques.

    C'est BLOQUANT : join ne return pas tant que le thread tourne.

─── main() ───  ligne 60

    int     main(int argc, char **argv)

    Le flow complet :
    1. Verifier qu'on a 9 arguments (programme + 8 args = 9)
    2. Parser les arguments → remplir la struct sim
    3. Init la simulation → alloc memoire, init mutex
    4. Set le temps de depart (le plus tard possible avant les threads)
    5. Lancer les threads → la simulation demarre !
    6. Joindre les threads → attendre la fin
    7. Detruire la simulation → free tout
    8. Return le code de retour (0 = succes, 1 = erreur)

─── update_last_compile() ───  ligne 15

    void    update_last_compile(t_coder *c)
    {
        pthread_mutex_lock(&c->mtx);
        c->last_compile = now_ms();
        pthread_mutex_unlock(&c->mtx);
    }

    Enregistre "je viens de commencer a compiler maintenant".
    Le mutex c->mtx protege l'ecriture parce que le MONITOR lit
    last_compile en parallele (dans check_burnouts).

    Si on ecrivait sans mutex : le monitor pourrait lire une valeur
    "a moitie ecrite" (sur certaines architectures, ecrire un long
    de 8 octets n'est pas toujours atomique).

─── coder_routine() ───  ligne 49  ★ LA FONCTION LA PLUS IMPORTANTE ★

    void    *coder_routine(void *arg)

    C'est LA BOUCLE DE VIE d'un coder. Chaque thread execute cette fonction.

    Signature : void *f(void *arg) → impose par pthread_create.
    Le void* est caste en t_coder* a la premiere ligne.

    La boucle :

    while (!sim_stopped(c->sim))     ← check si la simu est finie
    {
        === PHASE 1 : PRENDRE LES DONGLES ===

        if (take_dongles(c))         ← essaie de prendre 2 dongles
            break ;                  ← si echec (simu arretee), on sort

        === PHASE 2 : COMPILER ===

        update_last_compile(c);      ← reset le chrono burnout
        log_state(c, "is compiling");
        precise_sleep(t_compile);    ← dort pendant le temps de compile
        release_dongles(c);          ← repose les 2 dongles sur la table
        inc_compiles_done(c);        ← +1 au compteur de compiles

        === PHASE 3 : DEBUGGER ===

        if (sim_stopped(c->sim))     ← re-check avant de continuer
            break ;
        log_state(c, "is debugging");
        precise_sleep(t_debug);

        === PHASE 4 : REFACTORER ===

        if (sim_stopped(c->sim))
            break ;
        log_state(c, "is refactoring");
        precise_sleep(t_refactor);

        → retour au debut de la boucle (while)
    }
    return (NULL);

    POINTS CRITIQUES :

    1. update_last_compile AVANT log_state("is compiling")
       → le monitor voit le nouveau chrono LE PLUS TOT POSSIBLE
       → reduit le risque de faux burnout

    2. Les "sim_stopped" entre chaque phase
       → si le monitor detecte un burnout pendant le debug, le coder
         sort au prochain check au lieu de continuer en refactoring

    3. Le "break" si take_dongles echoue
       → ca veut dire soit la simu est arretee, soit on est en N=1
       → dans les deux cas, on sort proprement

─── get_dongle_ids() ───  ligne 15

    static void get_dongle_ids(t_coder *c, int *first, int *second)

    Calcule QUELS dongles le coder doit prendre, et dans QUEL ORDRE.

    Table ronde circulaire :
    - Coder d'id=i (1-indexe) a :
      - dongle gauche = i-1   (index 0-base)
      - dongle droit  = i % N (index 0-base)

    Exemple avec 5 coders :
    Coder 1 : left=0, right=1 → first=0, second=1
    Coder 2 : left=1, right=2 → first=1, second=2
    Coder 3 : left=2, right=3 → first=2, second=3
    Coder 4 : left=3, right=4 → first=3, second=4
    Coder 5 : left=4, right=0 → first=0, second=4  ← INVERSION !

    Le coder 5 prend dongle 0 AVANT dongle 4 (order croissant).
    C'est CA qui casse le circular wait et empeche le deadlock.

    Sans cette inversion : tout le monde prend son gauche en premier →
    tout le monde tient 1 dongle → tout le monde attend l'autre →
    DEADLOCK. Personne ne bouge plus.

    Avec l'inversion : le coder 5 essaie de prendre dongle 0 d'abord.
    Mais coder 1 veut aussi dongle 0. L'un des deux attend, l'autre
    avance. Pas de cycle. Pas de deadlock.

    Cas special N=1 : left=0, right=0 → first = second = 0.
    Le coder a besoin du MEME dongle deux fois → impossible.

─── wait_for_stop() ───  ligne 34

    static int  wait_for_stop(t_coder *c)

    Cas N=1 : le coder ne peut jamais compiler (il lui faut 2 dongles,
    il n'y en a qu'un). Il attend simplement que le monitor detecte
    le burnout et arrete la simulation. Retourne 1 (echec).

─── take_dongles() ───  ligne 41

    int     take_dongles(t_coder *c)

    Prend les 2 dongles du coder. Retourne 0 si OK, 1 si echec.

    Flow :
    1. Calcule first et second (en ordre anti-deadlock)
    2. Si first == second → N=1, wait_for_stop
    3. Acquire le PREMIER dongle
       → Si echec (simu arretee) : return 1
       → Si OK : log "has taken a dongle"
    4. Acquire le SECOND dongle
       → Si echec : RELEASE le premier (sinon il reste bloque!) + return 1
       → Si OK : log "has taken a dongle"
    5. Return 0 (les deux sont pris)

    SUPER IMPORTANT : si l'acquisition du 2eme echoue, on DOIT
    release le 1er. Sinon il reste in_use pour toujours et les autres
    coders sont bloques dessus.

─── release_dongles() ───  ligne 61

    void    release_dongles(t_coder *c)

    Repose les 2 dongles. Appelle release_dongle pour chacun.
    L'ordre de release n'a pas d'importance (pas de risque de deadlock
    en relachant des ressources).


################################################################################
#                                                                              #
#   FICHIER 9 : srcs/dongle_acquire.c  (le mecanisme d'acquisition)           #
#                                                                              #
################################################################################

C'est LE FICHIER LE PLUS COMPLEXE. C'est ici que la concurrence,
le scheduler, le cooldown, et la priority queue se rencontrent.

─── make_request() ───  ligne 15

    static t_request    make_request(t_coder *c)

    Cree un "ticket" pour la file d'attente du dongle.

    - req.coder_id = c->id    → qui demande
    - req.key :
      - FIFO : now_ms()       → timestamp d'arrivee (premier arrive = petite key)
      - EDF  : get_last_compile(c) + sim->t_burnout
                               → deadline du coder (plus urgent = petite key)

    C'est ICI que FIFO et EDF different. La priority queue ne sait pas
    ce que la key SIGNIFIE — elle trie juste par plus petite key en premier.

─── can_take() ───  ligne 27

    static int  can_take(t_dongle *d, int coder_id)

    Verifie si le coder PEUT prendre le dongle maintenant.
    3 conditions TOUTES requises :

    1. !d->in_use             → personne ne le tient
    2. top->coder_id == id    → c'est MON tour dans la file (je suis en tete)
    3. now >= released_at + cooldown → le cooldown est fini

    Si UNE condition est fausse → return 0 → le coder continue d'attendre.

    Pourquoi checker la tete de la pq ? C'est le SCHEDULER.
    Meme si le dongle est libre, si un autre coder avec plus haute
    priorite attend aussi, JE NE PASSE PAS. C'est la fairness.

─── timed_wait() ───  ligne 41

    static void timed_wait(t_dongle *d)

    Fait dormir le thread en attendant un signal sur la cond var du dongle.

    Calcul du timeout :
    - gettimeofday → heure actuelle
    - on ajoute 10ms (10000 microsecondes)
    - conversion en struct timespec (secondes + nanosecondes)
    - si les nanosecondes depassent 1 milliard (1 seconde), on ajuste

    pthread_cond_timedwait fait 3 choses ATOMIQUEMENT :
    1. LIBERE le mutex d->lock (pour que d'autres threads puissent release)
    2. DORT jusqu'a un signal (broadcast) ou le timeout
    3. RE-PREND le mutex d->lock au reveil

    C'est crucial : pendant que le coder dort, le mutex est LIBRE.
    Un autre coder peut donc appeler release_dongle (qui lock le mutex,
    set in_use=0, broadcast, unlock). Le broadcast reveille notre coder.

    Le timeout de 10ms garantit que meme sans signal, le coder se
    reveille et re-check can_take() et sim_stopped(). C'est un filet de
    securite.

─── acquire_dongle() ───  ligne 57  ★ FONCTION CRITIQUE ★

    int     acquire_dongle(t_dongle *d, t_coder *c)

    Acquiert UN dongle. Bloque jusqu'a ce que le coder l'obtienne ou que
    la simu s'arrete.

    Flow detaille :

    1. make_request(c)           → cree le ticket
    2. pthread_mutex_lock        → prend le verrou du dongle
    3. pq_push                   → s'inscrit dans la file d'attente
    4. WHILE (!can_take)         → boucle d'attente
       a. sim_stopped ?          → si oui : se retire de la pq, unlock, return 1
       b. timed_wait             → dort 10ms max (libere le mutex pendant)
       c. reveil → re-check can_take (retour en haut du while)
    5. pq_pop                    → se retire de la tete de la file
    6. d->in_use = 1             → marque le dongle comme pris
    7. pthread_mutex_unlock      → libere le verrou
    8. return 0                  → succes

    POURQUOI pq_push AVANT le while et pq_pop APRES ?
    → Le coder doit etre DANS la file pour que can_take le voie en tete.
    → Il doit rester dans la file pendant qu'il attend.
    → Il se retire quand c'est son tour et qu'il prend effectivement le dongle.

    POURQUOI pq_remove en cas de sim_stopped ?
    → Si le coder part sans se retirer de la pq, les autres coders
    verront un "fantome" en tete de file et personne ne pourra prendre
    le dongle.

─── release_dongle() ───  ligne 80

    void    release_dongle(t_dongle *d)

    Repose le dongle sur la table.

    1. Lock le mutex du dongle
    2. in_use = 0            → le dongle est libre
    3. released_at = now_ms() → le cooldown demarre MAINTENANT
    4. pthread_cond_broadcast → reveille TOUS les coders qui dorment
       sur ce dongle (dans timed_wait)
    5. Unlock

    Pourquoi BROADCAST et pas SIGNAL ?
    - signal reveille UN SEUL thread. Mais lequel ? Aleatoire.
    - Si le thread reveille n'est pas en tete de la pq, il se rendort,
      et le vrai "prochain" reste endormi.
    - broadcast reveille TOUT LE MONDE. Chacun check can_take, et seul
      celui en tete (+ dongle libre + cooldown ok) passe. Les autres
      se rendorment.
    - C'est un peu plus de CPU mais c'est CORRECT.


################################################################################
#                                                                              #
#   FICHIER 10 : srcs/priority_queue.c  (le min-heap)                         #
#                                                                              #
################################################################################

Un MIN-HEAP est un arbre binaire ou chaque parent est PLUS PETIT que ses
enfants. L'element le plus petit est toujours a la racine (index 0).

On le stocke dans un TABLEAU PLAT. Pas besoin de pointeurs gauche/droit.

    Index :    0
              / \
             1   2
            / \ / \
           3  4 5  6

    Formules :
    parent(i) = (i - 1) / 2
    gauche(i) = 2 * i + 1
    droite(i) = 2 * i + 2

─── pq_init() ───  ligne 15

    int     pq_init(t_pq *pq, int cap, int (*cmp)(t_request, t_request))

    - malloc un tableau de "cap" requests (cap = nombre de coders)
    - size = 0 (vide au debut)
    - cmp = la fonction de comparaison

    Pourquoi cap = nombre de coders ? Parce qu'au maximum, N coders
    peuvent attendre le meme dongle en meme temps.

─── pq_push() ───  ligne 26

    void    pq_push(t_pq *pq, t_request req)

    Ajoute un element :
    1. Si plein → return (ne devrait jamais arriver)
    2. Place le nouvel element a la FIN du tableau
    3. size++
    4. sift_up → remonte l'element jusqu'a ce qu'il soit a sa place

    Complexite : O(log N) — au pire, l'element remonte de la feuille
    jusqu'a la racine.

─── pq_pop() ───  ligne 35

    t_request   pq_pop(t_pq *pq)

    Retire et retourne l'element de plus haute priorite (index 0).

    1. Sauvegarde data[0] (le minimum)
    2. size--
    3. Si il reste des elements : met le DERNIER element a l'index 0
    4. sift_down → descend cet element jusqu'a sa bonne place
    5. Retourne l'ancien minimum

    Complexite : O(log N)

─── pq_peek() ───  ligne 49

    t_request   *pq_peek(t_pq *pq)

    Retourne un POINTEUR vers l'element en tete, sans le retirer.
    Si la pq est vide → NULL.
    Utilise par can_take pour verifier "est-ce que JE suis en tete ?"

─── pq_destroy() ───  ligne 56

    void    pq_destroy(t_pq *pq)

    Free le tableau et reset. Appele pendant destroy_sim.

#                                                                              #
################################################################################

─── pq_swap() ───  ligne 15

    static void pq_swap(t_pq *pq, int a, int b)

    Echange deux elements dans le tableau. Classique.

─── sift_up() ───  ligne 24

    void    sift_up(t_pq *pq, int i)

    Remonte l'element a l'index i vers la racine.

    while (i > 0)                           → tant qu'on n'est pas a la racine
        parent = (i - 1) / 2               → calcule l'index du parent
        if (!cmp(data[parent], data[i]))    → si parent <= enfant, on arrete
            break
        pq_swap(pq, i, parent)              → sinon, on echange
        i = parent                          → et on continue vers le haut

    Comment lire le cmp :
    cmp(a, b) retourne 1 si "a devrait etre EN DESSOUS de b dans le heap".
    Donc cmp(parent, child) = 1 signifie "parent est trop gros, swap !".
    cmp(parent, child) = 0 signifie "parent est bien place, stop".

─── sift_down() ───  ligne 38

    void    sift_down(t_pq *pq, int i)

    Descend l'element a l'index i vers les feuilles.

    while (1)
        min = i                        → commence par supposer que i est le min
        l = 2*i + 1, r = 2*i + 2      → enfants gauche et droit
        si l existe ET data[min] > data[l] → min = l
        si r existe ET data[min] > data[r] → min = r
        si min == i → i est bien place, break
        sinon → swap avec min, continuer

    On compare AVEC LES DEUX ENFANTS et on swap avec le plus petit.
    Ca garantit que la propriete min-heap est restauree.

─── cmp_request() ───  ligne 62

    int     cmp_request(t_request a, t_request b)

    Retourne 1 si a devrait etre APRES b (a est moins prioritaire).

    if (a.key == b.key)           → en cas d'egalite
        return (a.coder_id > b.coder_id)  → plus petit id passe d'abord
    return (a.key > b.key)        → plus petite key = plus prioritaire

    Cette fonction est la MEME pour FIFO et EDF. La difference est dans
    le CALCUL de la key (fait dans make_request), pas dans la comparaison.

─── pq_remove() ───  ligne 69

    void    pq_remove(t_pq *pq, int coder_id)

    Retire un coder specifique de la pq (utilise quand sim s'arrete).

    1. Cherche lineairement l'element avec le bon coder_id
    2. Le remplace par le DERNIER element du tableau
    3. size--
    4. Restaure la propriete du heap avec sift_down PUIS sift_up
       (un des deux sera un no-op, on ne sait pas lequel a l'avance)

─── check_all_done() ───  ligne 15

    static int  check_all_done(t_sim *sim)

    Parcourt tous les coders. Si TOUS ont compiles_done >= compiles_req,
    retourne 1. Sinon retourne 0.

    Utilise get_compiles_done() → chaque lecture est protegee par c->mtx.

─── check_burnouts() ───  ligne 28

    static int  check_burnouts(t_sim *sim)

    Parcourt tous les coders. Pour chacun :
    - since = now_ms() - get_last_compile(c) → temps depuis dernier compile
    - si since >= t_burnout → ce coder a burnout !
    - retourne son ID (1..N)

    Si personne n'a burnout → retourne 0.

    Retourne le PREMIER coder qui a burnout. S'il y en a plusieurs en
    meme temps, seul le premier est logue (les autres sont coupes par
    le flag stop).

─── log_burnout() ───  ligne 43

    static void log_burnout(t_sim *sim, int id)

    Logue "burned out" et arrete la simulation.

    1. Lock log_mtx
    2. if (!stop) → verifie qu'on n'a pas deja stoppe
    3. printf("... burned out\n")
    4. stop = 1
    5. Unlock

    Le check "if (!stop)" sous le lock empeche de logger 2 fois "burned out"
    si deux coders burnoutent au meme instant.

    Le printf ET le set de stop sont sous LE MEME LOCK. Ca garantit
    qu'aucun autre log ne passe entre "burned out" et stop=1.

─── monitor_routine() ───  ligne 55  ★ LE GARDIEN ★

    void    *monitor_routine(void *arg)

    Boucle infinie :
    1. check_all_done → si oui : set stop, return (fin propre)
    2. check_burnouts → si oui : log_burnout, return (fin avec mort)
    3. usleep(500) → dort 0.5ms, puis recommence

    Pourquoi 500 microsecondes ? C'est le compromis :
    - Plus court (100us) → detecte plus vite mais bouffe du CPU
    - Plus long (5000us) → economise le CPU mais detecte trop tard (>10ms)
    - 500us → environ 2000 checks par seconde, precision largement sous 10ms

    L'ordre (all_done AVANT burnouts) est delibere :
    → Si tous les coders ont fini ET qu'un coder est pile sur sa deadline,
      on considere que c'est un SUCCES, pas un burnout.

    Le "return (NULL)" final apres le while(1) est du code mort.
    Mais le compilateur pourrait raler "control reaches end of non-void
    function" sans lui. Donc on le garde.

#   LES MUTEX ET CE QU'ILS PROTEGENT                                          #

################################################################################

    MUTEX               PROTEGE                      QUI L'UTILISE
    ─────────────────────────────────────────────────────────────────
    sim->log_mtx        sim->stop + stdout (printf)  Tous les threads
    c->mtx              c->last_compile              Coder + Monitor
                        c->compiles_done             Coder + Monitor
    d->lock             d->in_use                    Coders
                        d->released_at               Coders
                        d->waiters (la pq)           Coders
                        d->cond (lie au lock)        Coders

    ORDRE DES LOCKS (pour eviter deadlock entre mutex) :
    1. d->lock (dongle) — pris en premier si necessaire
    2. c->mtx (coder) — jamais pris en meme temps que d->lock
    3. sim->log_mtx — peut etre pris en tenant d->lock

    Aucun thread ne prend DEUX mutex "de meme niveau" en meme temps,
    SAUF pour les dongles (mais l'ordre des IDs empeche le deadlock).


################################################################################
#                                                                              #
#   POURQUOI IL N'Y A PAS DE DEADLOCK                                         #
#                                                                              #
################################################################################

    Il y a 4 conditions pour un deadlock (Coffman). Si on en casse UNE,
    pas de deadlock.

    On casse le CIRCULAR WAIT (attente circulaire) :

    Regle : un coder prend TOUJOURS le dongle de plus petit ID d'abord.

    Preuve par contradiction :
    - Suppose un cycle : A attend D1 (tenu par B), B attend D2 (tenu par C),
      C attend D3 (tenu par A).
    - A tient D3 et attend D1 → D3 > D1 (il a pris le plus petit d'abord)
    - B tient D1 et attend D2 → D1 > D2
    - C tient D2 et attend D3 → D2 > D3
    - Donc D3 > D1 > D2 > D3 → CONTRADICTION.
    - Le cycle est impossible. CQFD.
